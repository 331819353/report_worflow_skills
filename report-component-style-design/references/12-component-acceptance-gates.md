# Component Acceptance Gates

Detailed gates moved out of `SKILL.md` to keep the component skill lightweight. Load this only when producing or judging implementation-ready component specs.

## Hard Constraints

- Do not style a component until its business purpose, data grain, key fields, filter scope, and interaction state are known.
- Do not accept an implementation-ready component spec that lacks a `Positioning And Alignment Rules` / `位置排布与对齐规则` chapter with container variables, slot x/y positions, alignment, size tiers, and state geometry.
- Do not accept a component that has component-internal filters but lacks a `组件内筛选区 / 局部筛选区` section. It must define suitable filters, unsuitable filters, capsule/dropdown style, placement, size algorithm, title/unit/legend/metric relationship, and responsive collapse.
- Component-internal local filters affect only the current component or declared local group. They must not change page/global scope, permission scope, backend aggregation, pagination, export scope, or other components.
- Component-internal local filters use capsule sliding buttons for `2-4` short options, collapse to compact dropdown when options exceed `4` or width fails, and use a panel/popover/drawer trigger only for multiple local filter groups.
- Component-internal local filters must not cover or compress KPI values, chart plots, legends, axes, table headers, pagination, or state messages. Collapse the filter, secondary metrics, or actions before shrinking the core value/plot/table below its budget.
- Analysis & Insight components are a distinct report component family, even when implemented with `visualType: 'text-summary'`. A valid analysis/explanation component declares `analysisInsightContract` or equivalent metadata with subtype, conclusion, evidence, affected object, comparison/change value when relevant, action or drill path, confidence/definition/source/freshness when relevant, local-filter scope, tooltip payload, size tier, and states. Do not accept generic insight copy, essay-like explanations, full-card red warnings, decorative icons, or a card that says "发生变化/建议关注" without business evidence.
- Composite Panels / multi-component analysis cards are valid only when multiple child components answer one business question as one mini analysis loop: summary -> trend/structure -> contribution/exception -> detail/action. A valid Composite Panel declares `compositePanelContract`, one primary visual, default `2-3` child components, maximum `4` visible analytical child components unless a documented large-container exception exists, shared local filter scope, shared legend/unit behavior, child priority, linked hover/click context, child minimum sizes, responsive fallback, and partial child-state rules. Do not use a Composite Panel as a miniature dashboard, equal-weight chart collage, nested-card stack, or workaround for page-grid placement.
- Do not treat CSS flex/grid auto layout as sufficient by itself for dense report components. It must be backed by a measurable slot budget and a fallback when labels, values, icons, legends, controls, or state messages do not fit.
- Every implementation-ready component must declare a display budget for its family and a specific overflow strategy. Missing limits for text-summary items, cards/list rows, journey steps, KPI peers, chart categories/series/labels, table columns/rows, local-filter options, nodes/links, hierarchy depth, or annotations are acceptance gaps.
- Do not use generic "AI/SaaS polish" as a component style: purple-blue gradients, glass panels, black neon, glow buttons, floating decorative cards, oversized radius, abstract AI icons/3D assets, or particle/grid decoration require a named brand/template/sample rationale.
- Component copy must be concrete. Titles, subtitles, CTAs, empty states, errors, and helper text must name the user action, data object, condition, owner, source, period, or next step. Generic words such as "赋能", "智能化", "一站式", "重新定义", "无缝", or "提升效率" are not enough.
- KPI cards, metric groups, chart blocks, and summary components must not be styled as generic metric shells. Every primary metric-bearing component must preserve visible or discoverable formula/denominator, grain, period, source/freshness, unit/precision, baseline, and owner/action context when the surface is implementation-ready.
- Do not add chart variety for decoration. A chart/table choice must serve a specific task such as target gap, trend, ranking, composition, distribution, relationship, driver diagnosis, abnormality, detail, or action. Otherwise record `RPT-DECORATIVE-CHART`.
- Visual treatment must not compete with data reading. Gradients, glass, glow, illustration, large shadows, and high-saturation decorative colors fail when they reduce numeric hierarchy, axis/legend readability, anomaly visibility, table scanning, or exact-value access.
- Report components that show a result must expose the next step or evidence path: tooltip, detail table, drawer, drilldown, export, owner/action, annotation, or runbook. A polished result card with no action path is `RPT-NO-ACTION`.
- Prototype or mock component states must not imply unrealistically clean performance. All-green/all-up KPIs, perfectly smooth trends, balanced shares, and no missing/zero/extreme states should be marked `RPT-TOO-CLEAN-DATA` unless the source evidence proves that reality.
- Icon-only and decorative components are not accepted when they carry decision-critical meaning without labels, tooltips, data, or accessible names.
- Motion must communicate state, feedback, hierarchy, loading, selection, or transition. Decorative movement that distracts from scanning or has no reduced-motion behavior fails component style QA.
- Do not accept a polished component that is unreasonable for the task. Use `DESIGN-*` findings when a chart should be a table, a dense component needs drilldown/fullscreen, a component duplicates another message, or the style hides the user's decision-critical value.
- Do not hide decision-critical labels, units, warnings, or values without a hover/focus/click disclosure path.
- Do not let ECharts, S2, SVG, canvas, or custom diagrams mount into a zero-size or unstable parent block or sub-block container.
- When a component is a standard ECharts chart, implement it as an ECharts chart: project wrapper or `echarts.init`, data-driven `option`/`series`, normal `setOption`/update/resize, and ECharts tooltip/legend/emphasis behavior. Do not import ECharts while hand-drawing bars, lines, pies, gauges, maps, parallel-coordinate axes/lines, sunburst arcs, relation graph nodes/edges, axes, or legends with SVG/HTML/CSS/canvas.
- When a component is restored from HTML/source, do not treat sample `<svg>`, `<canvas>`, CSS bars, DOM arcs, or absolute-positioned chart marks as reusable chart code. They are visual/config evidence only. Standard chart components must be rebuilt through ECharts data-driven options/series unless a named custom-diagram exception is declared before implementation.
- Hand-authored SVG/canvas is allowed only for icons, logos, trend arrows, decorative assets, or explicitly named custom diagrams/graphics that are not standard charts. ECharts output using `renderer: 'svg'` is valid because ECharts owns the generated SVG.
- Do not stretch SVG, canvas, ECharts custom graphics, gauges, maps, radar, sunburst, flow paths, or generated decorative geometry with independent X/Y scaling. Preserve geometry with a canonical `viewBox`, `preserveAspectRatio="xMidYMid meet"`, CSS `aspect-ratio`, or a measured inner fit box using `scale = min(containerWidth / designWidth, containerHeight / designHeight)`.
- Do not use CSS `transform: scaleX/scaleY`, `object-fit: fill`, or raw `width: 100%; height: 100%` SVG/canvas stretching when the graphic shape has business meaning. Center/letterbox inside the assigned viewport or redesign the graphic for the actual aspect ratio.
- Radar charts must declare dimension count, visible series count, shared score scale, standardization rule for mixed raw units, radius/center, ring count, dimension-label gap, and value-label strategy before styling. Do not accept radar with `> 10` dimensions, `> 3` visible series, mixed raw units plotted directly, or an elliptical/stretched coordinate system.
- Scatter/bubble charts must declare X metric, Y metric, units, axis range/baseline behavior, point count, point density strategy, point or bubble size mapping, label strategy, reference-line/quadrant behavior, legend separation, and tooltip payload before styling. Do not accept scatter with only one numeric metric, missing X/Y units, dense points without sampling/aggregation/zoom, all-point labels in non-sparse charts, or unmanaged DOM overlays for points/axes/reference lines.
- Parallel coordinates must declare the multi-metric object comparison task, object/sample id/name fields, `3-12` dimension fields, dimension order, per-axis unit/range/scale direction, standardization or independent-axis rule, sample count, line opacity tiers, highlight/group semantics, axis spacing budget, brush behavior when present, legend/filter separation, tooltip payload, and table/scatter/bar fallback before styling. Do not accept parallel coordinates for `1-2` metrics, single ranking, exact value audit, `>12` dimensions without filtering/scroll, excessive sample lines without opacity/sampling/aggregation, or unmanaged hand-drawn axes/lines when ECharts parallel is claimed.
- Map/geographic charts must declare the geography decision task, region code or longitude/latitude fields, map resource/projection, fitBounds/aspect-ratio behavior, map layer type, visualMap/legend strategy, point/flow density, bubble or flow-size mapping when present, label strategy, missing-geo handling, and tooltip payload before styling. Do not accept maps with no real geography field, purely decorative basemaps, stretched geography, all-region/all-point labels, dense points/flows without clustering/heatmap/TopN, or unmanaged hand-drawn map layers when ECharts map is claimed.
- Candlestick/K-line charts must declare ordered time, `open/high/low/close`, unit, market rise/fall color convention, price range padding, visible candle count, candle/body width, main/volume/indicator height budget, MA/indicator count, axis label sampling, crosshair/tooltip payload, dataZoom behavior, and missing OHLC/volume handling before styling. Do not accept candlestick charts with only one value per time point, unsorted OHLC rows, invalid OHLC relationships, dense candles without dataZoom/recent-window fallback, overloaded technical indicators, or unmanaged hand-drawn candles when ECharts candlestick is claimed.
- Boxplot charts must declare raw-sample or precomputed statistical contract, sample count, Q1/median/Q3/IQR, whisker and outlier rule, unit, axis range, category count, grouped-series count, box/whisker geometry, outlier density strategy, label strategy, and tooltip payload before styling. Do not accept boxplots for single aggregate ranking, sample-too-small categories without fallback, unspecified outlier rules, grouped series beyond `S > 3` without scroll/fallback, all-stat labels on every category, or unmanaged hand-drawn boxes when ECharts boxplot is claimed.
- Matrix/time/calendar/correlation heatmaps must declare row dimension, column dimension, value metric, aggregation grain, unit, color-scale type, visualMap range, missing-vs-zero encoding, row/column count, cell size, label sampling, value-label threshold, highlight rule, and tooltip payload before styling. Do not accept non-geographic heatmaps for one-dimensional ranking, exact small-number comparison, tiny value variance, unbounded row/column cardinality, unclear color semantics, all-cell permanent labels, missing values rendered as zero, or unmanaged hand-drawn heatmap cells when ECharts heatmap/calendar heatmap is claimed.
- Detail Tables must declare row-level business task, row grain, primary key/object field, visible column priorities, column type/width/alignment metadata, default sort, local/global filter behavior, search/sort/pagination/export execution scope, row action/detail contract, fixed-header/frozen-column conditions, row-height density, table-body height budget, and empty/loading/error/no-permission states before styling. Default visible columns are `5-8`, large tables may show `8-12`; more columns require horizontal scroll, frozen primary/action columns, column settings, drawer/detail, grouping, or split. Do not accept a dense source-table dump, every-column sorting, checkbox columns without batch action, many inline row actions, client-only global pagination/sort/search over unbounded data, or table bodies with fewer than `3` visible rows unless the component is explicitly a small preview with a detail route.
- Pivot Tables must declare the multidimensional cross-summary task, row dimensions, column dimensions, measures, aggregation grain, measure formulas/aggregation functions, subtotal/grand-total rules, row/column hierarchy depth, row/column sorting, fixed header/frozen row dimension behavior, `pivotAreaH >= CH * 0.55`, row/column density limits, tooltip/drilldown payload, and loading/empty/error/no-permission states before styling. Use AntV S2 or a project S2-equivalent renderer for real pivot/cross/wide analytical tables. Do not accept a raw record list disguised as a pivot, an Excel-like field well in report display mode, naive sum/average for percentage metrics, all-cell over-coloring, or hand-rolled merged-header tables when S2 behavior is required.
- Grouped/complex table headers must be declared whenever a table has more than `8` visible columns or natural field groups such as metric families, actual/target/variance, current/YoY/MoM, amount/count/rate, time periods, financial sections, or pivot columns. A valid grouped header includes a `columnTree` or equivalent grouped-column contract, leaf field metadata, unit/definition metadata, `colSpan`/`rowSpan` calculation, parent-width-from-leaf-width rules, maximum depth `<=3` by default, fixed-header/frozen-column behavior, component-local filter vs per-column filter separation, sort/filter icon limits, tooltip definitions, density fallback, and loading/empty/error/no-permission geometry. Do not fake complex headers with absolute overlays, strong decorative color blocks, every-column icons, or header height that leaves too few body rows.
- Combo charts must declare the paired business relationship, primary bar metric, secondary line/target metric, category/time grain, left/right y-axis unit mapping, total series count, dual-axis rationale when used, category count/density, plot-height budget, legend/filter separation, tooltip payload, and split-chart fallback before styling. Default Combo is one bar series plus one line or target/reference series. Do not accept unrelated metrics, more than `4` total visible series, hidden/ambiguous right axes, false dual-axis correlation, dense category labels without sampling/scroll, or hand-authored bar/line/axis marks while ECharts combo is claimed.
- Dual-axis ECharts Combo charts with a top legend must declare a top-band collision budget: final `legend.top/left/right`, `grid.top`, both y-axis `name` values, `nameLocation`, `nameGap`, `nameRotate` when used, `nameTextStyle`, axis-label unit strategy, and measured legend/axis-name rectangles. `grid.containLabel` is not enough. Do not accept fixed `legend.top/right` and `grid.top` numbers unless legend text/marker boxes and left/right y-axis name boxes keep at least `8px` safe gap with no overlap at every target viewport.
- Gauge charts must declare a single bounded progress/status metric, min/max range, unit, current value, clamp/overflow behavior, target/threshold semantics when present, business direction for status color, gauge type, angle span, radius/arc width, center value and unit placement, tick/label density, component-local filters, tooltip payload, and KPI/bar/line/table fallback before styling. Default report Gauge is a semicircle with center value and light target marker. Do not accept gauges for multi-category comparison, trend, composition, unbounded metrics, exact-value audit, many gauges in one screen, decorative mechanical dials, or hand-drawn needles/arcs while ECharts gauge is claimed.
- Path/user/process path charts must declare path task, step/node schema, directed transition/link schema, start node, end node, step order or layer, metric basis, conversion/drop-off formulas, path depth, Top N rule, node/link counts, main-path vs branch strategy, path-width mapping, label strategy, legend/filter separation, component-local filters, tooltip payload, and aggregation/fallback before styling. Do not accept path charts for simple ranking, composition, unordered entity relationships, geographic routes, all-branch expansion, unbounded paths, all-link labels, or decorative journey lines with no transition evidence.
- Funnel charts must declare a real ordered conversion/process task, stage dataset/schema, stage order, metric basis/unit, shared population or documented cohort logic, entry/final values, entry share, stage conversion, drop value/rate, total conversion, target/comparison handling when present, stage density, label strategy, component-local filters, tooltip payload, and table/path/Sankey/bar fallback before styling. Default to a horizontal bar funnel for report components; use trapezoid funnels only for large-display storytelling with few stages. Do not accept funnel charts for unordered ranking, time trend, part-to-whole composition, multi-branch paths, arbitrary decreasing categories, mixed people/order/amount stages, all-loss labels, hidden exact values, or decorative hand-drawn funnels while ECharts/bar-driven charting is claimed.
- Sankey diagrams must declare the flow task, node schema, directed link schema with `source`/`target`/`value`, layer/stage order, metric unit, node-value rule, flow-width mapping, direction, Top N/`其他` aggregation, node/link density limits, label strategy, legend/filter separation, component-local filters, hover/click highlight semantics, tooltip payload, and table/path/funnel/bar fallback before styling. Do not accept Sankey for data without a real source-target flow, simple ranking, single-level composition, exact-row audit, negative flow values, unbounded node/link counts, or hand-drawn decorative ribbons while ECharts `sankey` is claimed.
- Treemap/rectangular tree maps must declare the hierarchical composition task, hierarchy fields, parent/leaf aggregation, non-negative additive area metric, optional color metric, total/parent share formulas, level depth, Top N/`其他` aggregation, node count, minimum rectangle and label thresholds, squarified/layout strategy, color legend, breadcrumb/drilldown behavior when deep, tooltip payload, and table/bar/tree fallback before styling. Do not accept treemap for trend, negative values as area, precise comparison of many similar values, unclear hierarchy, unaggregated high-cardinality leaves, or ordinary parent-child expansion tasks that need a tree.
- Sunburst charts must declare the hierarchy path and composition task, hierarchy fields or `children`, non-negative additive angle metric, parent/child angle mapping, total-share and parent-share formulas, visible depth/ring count, center radius and ring-width budget, Top N/`其他` aggregation, sector/label thresholds, center content, color semantics, breadcrumb/drilldown behavior, tooltip payload, and Treemap/bar/table fallback before styling. Do not accept sunburst for single-level categories, trend, precise ranking, negative/rate/score angle values, `> 4` visible levels without drilldown, too many siblings without Top N, or decorative multi-ring pies with thin business context.
- Tree/hierarchical tree charts must declare hierarchy task, tree data schema, root node, parent-child relationship fields, node id/name/type/status/value, depth, layer order, visible depth, default expanded levels, child-count and Top N/aggregation rules, orientation, node/link density, expand/collapse behavior, search/locate behavior when dense, label strategy, legend/filter separation, tooltip payload, and tree-list/fallback rules before styling. Do not accept tree charts for many-to-many relationship graphs, composition-by-area tasks that need Treemap, simple ranking, or full expansion of all nodes without collapse/search.
- Relation/network graphs must declare entity node fields, relationship edge fields, graph task, layout type, node/edge counts, node categories, edge types/directions/weights, density strategy, node-size and edge-width mapping, label strategy, legend/search/control separation, fitView/zoom/drag behavior, and node/edge tooltip payload before styling. Do not accept relation graphs for simple ranking/trend/detail-table tasks, unbounded node or edge counts, dense hairball layouts without filtering/aggregation/local exploration, all-node or all-edge labels, mixed graph legend and filters, or unmanaged hand-drawn graph marks when ECharts graph is claimed.
- For ECharts line, area, bar, and other category-axis charts, the x-axis categories must be ordered explicitly. Time/period axes default to ascending chronological order unless the user or business rule explicitly requires reverse/custom order.
- Do not sort only x-axis labels, categories, or `xAxis.data` while series values still map the unsorted source rows. Sort the row tuples first, then derive `xAxis.data`, every `series.data`, tooltip payloads, and click payloads from the same ordered rows.
- ECharts stacked bar/column charts must preserve segment continuity: all touching segments share one `stack` and category index, internal joins have square corners, only outer stack ends may be rounded, segment borders default to `0`, and no background sliver, overlap notch, or rounded internal seam appears in screenshot crops. Do not accept full-pill radius on every stacked series.
- Line/area charts must declare x-axis order, period granularity, y-axis baseline/range behavior, point density, label density, missing-value behavior, and target/average/warning line behavior before styling.
- Do not silently convert missing/null line-chart values to `0`. Missing points must break the line or show an explicit gap; real zero values must still render as `0`.
- Target/actual bar or column charts must declare target encoding before styling: unified target line, per-category target tick, or grouped actual/target bars. The choice must match whether targets are shared or category-specific and whether exact gap comparison is the task.
- For ECharts Cartesian charts with visible x-axis labels and a bottom legend, `grid.containLabel` must be `true`, `grid.bottom >= 56px`, and the legend must keep an explicit safe distance from x-axis labels.
- For donut/pie charts inside small cards or compact sub-blocks, bottom legend is the default. When outside labels are enabled, a right-side legend is allowed only after a width budget proves the card is wide enough; otherwise use a bottom legend.
- Donut/pie space budgets must declare `legendBandHeight`, `labelLineBudget`, `radius`, and `center` before acceptance. Right-side legends must also declare a legend width budget. Do not accept vague wording such as "adjust legend position" without these numbers/config fields.
- Right-side donut/pie legends are allowed only when the card is wide enough and outside labels are disabled or limited to key labels. Small-share categories may hide permanent outside labels; full category names, values, and percentages must remain available through tooltip and legend.
- Donut/pie labels, label lines, legends, center text, and card titles must not collide, squeeze each other, or leave the card bounds. Use label thresholds, `labelLayout.hideOverlap`, `bleedMargin`, `edgeDistance`, smaller radius, bottom legend, tooltip/legend disclosure, or a bar/table fallback before accepting the component.
- Pie/donut charts must declare category count, TopN/`其他` merge rule, negative/all-zero handling, legend position, center, radius, inner radius when donut, center metric, label strategy, and tooltip payload before styling. Prefer donut for report components; keep `2-6` categories by default and `<= 8` before merging.
- Do not accept pie/donut charts with negative values, fake all-zero shares, more than `8` unmerged categories, `其他` as the largest silent category, or outside labels/guide lines used as the default label surface in small/standard report cards.
- KPI cards must visually center the core value zone in the card body. The main value region must occupy at least 40% of the card's main visual height, and title/description text must not crowd the top while leaving a large empty lower area.
- KPI/metric cards must center `value + unit` as one value group. The unit follows the value with a 4-6px gap and baseline/bottom alignment; the card fails if the number is centered while the unit floats as a detached right-side label.
- KPI/metric cards must pass rendered bounding-box checks, not only CSS grid/flex row allocation. For default centered cards, `abs(valueGroupRect.centerY - valueAnchorViewport.centerY) <= 8px`; otherwise record `VIS-KPI-VALUE-OFFCENTER`. The value slot must own at least 40% of the main visual height, and the primary numeral glyph should normally occupy `22-28%` of the value anchor viewport for standard/enhanced primary cards. Do not accept `align-items: baseline` as the value slot's vertical placement strategy; baseline alignment is only allowed inside the centered value+unit group. For `W >= 360px` and `H >= 180px`, primary KPI values may use `40-44px` text after width/height fit proof. Auxiliary title/status/source/target content must collapse, move, or weaken before the primary value is made small or off-center.
- Tables with more than 8 visible columns or natural field groups must use complex/grouped headers by default. A flat header needs an explicit reason.
- Do not solve density by shrinking text below readable sizes; use sampling, scrolling, zoom/pan, drawer, fullscreen, split components, or table fallback.
- Do not create one-off component styles that conflict with the page shell, Haier branding, or existing design system.
- Do not duplicate Haier enterprise UI tokens or standard component rules inside report component guidance; load `$haier-enterprise-app-ui-design-spec` for Haier/enterprise report components as the inherited company baseline, then apply report component rules for analytical geometry, chart/table/KPI readability, and density.
- Do not render a duplicate visible component title inside the chart/table/KPI/custom component body when the surrounding block/container already provides a block-owned title. Component body titles are allowed only for explicitly standalone components without a block-owned title, and then title height must be reserved before measuring the viewport.
- KPI/metric card title ownership is explicit: `displayTitle` is the reader-facing block/card title, while `metricName` is the semantic metric identity for tooltip/export/口径/detail payload. When `normalized(blockTitle) === normalized(bodyMetricLabel)` or the two are highly similar, `showBodyMetricLabel` defaults to `false`; the body label moves to tooltip/definition/export metadata. If both remain visible in the same card without a disambiguating qualifier, record `VIS-DUPLICATE-TITLE`.
- Do not accept components that are too narrow, too small, or crowded for their labels, values, legends, axes, controls, pagination, and state messages. Enlarge the container, split the component, use scroll/zoom/fullscreen/drawer/table fallback, or reduce visible label density before final acceptance.
- Passing the outer `8 * N` parent block grid is not enough. Internal sub-blocks, composite widgets, summary areas, nested KPI grids, comparison tiles, and custom small cards must pass their own internal fit check inside the real sub-block/component viewport.
- Fixed-height cards, KPI tiles, navigation cards, compact summaries, and fixed-height control items must declare a height budget before component style acceptance: padding + explicit line-height rows + vertical gaps + badge/status/footer heights must be `<=` the component height. Auto layout alone is not a fit proof.
- Domain names, metric names, percentages/core values, KPI values, units, badges, helper text, and footer labels inside fixed-height components must have explicit `line-height`; large numbers need a deliberately sized line box.
- A sub-block is a local viewport inside a parent block. It may hold one component or one tightly related micro-group; it must have a declared min size, `5px` parent inset, `5px` sibling gap, overflow rule, and state behavior before component styling is accepted.
- For no-data masks inside composite parent blocks, component styling must respect block/container hierarchy: all sibling sub-blocks empty means one parent-block mask; partial empty means masks only on affected sub-blocks. A sub-block mask covers the sub-block title/label/control area and component body, not only the inner chart/table viewport.
- Do not force small metric groups into a fixed two-column grid when titles, values, units, or helper text cannot fit. Widen the summary zone, allow wrapping, increase min height, reduce visible tiles, stack, scroll, or move detail to tooltip/drawer.
- Do not combine `overflow: hidden`, `white-space: nowrap`, and `text-overflow: ellipsis` on decision-critical metric titles, summary labels, values, or action text unless a full text disclosure path is present. Prefer one- or two-line wrapping with stable height for long metric titles.
- For repeated peer cards/charts/tiles inside a component, sub-block group, or large parent block, use the internal exact `M * N` distribution only when `actualTotal > 4`; for `actualTotal <= 4`, use a small-group layout based on content and block shape. When the algorithm applies, calculate from `layoutTotal`: normally `layoutTotal = actualTotal`; when `actualTotal` is prime, use `layoutTotal = actualTotal + 1`; then choose columns `M >= N`, and the smallest `M - N` among valid factor pairs where `layoutTotal = M * N`. After the internal matrix is chosen, the parent block uses `heightExpansionRows = ceil(N * 2 / 3)` as the height expansion baseline; expand the parent block instead of squeezing child tiles. Do not add arbitrary empty slots; the only allowed spare cell is the single prime-balancing cell created when the algorithm applies to a prime count, and it must not create fake metrics or mock data. Split the group if the matrix becomes unreadable.
- Text summaries, conclusions, and insight blocks must inherit the surrounding page shell tokens for surface, spacing, typography, radius, and semantic colors. Do not introduce a new standalone visual surface, grid rhythm, or density that conflicts with adjacent sections.
- In `sampleRestore`, added conclusions must sit inside an existing sample-equivalent region and reuse its spacing, border, typography, and surface tokens; a new standalone horizontal band fails style QA unless the sample already has one.
- In fixed-grid report cards, KPI cards, chart/table containers, and compact controls, hover/focus feedback should use border color, outline, inset glow, or stable box-shadow. Do not use hover `translate`, `scale`, or outer-shadow effects when the parent container may clip borders/shadows or when the effect changes perceived block alignment.
- Hover/focus glow must not require extra layout space. Prefer `box-shadow: inset 0 0 0 1px ...`, subtle inner glow, or an absolutely positioned pseudo-element inside the component bounds. `focus-visible` should share the same non-shifting feedback as hover.
- Rate/change indicators in Chinese report UI display `%`, not `pt`, `p.p.`, or `percentage point`, unless the user explicitly requests that term.
- Change-rate and variance-rate indicators use positive-red-up / negative-green-down semantics with icon+text pairing: positive value = red text plus upward SVG/icon; negative value = green text plus downward SVG/icon; zero = neutral.
- Do not use naked native `<select>` controls as the final visual surface for primary filters.
- Do not hand-roll Vue controls that Element Plus already provides unless the existing project design system explicitly supersedes Element Plus or the interaction is unsupported.
- Do not accept funnel, flow, Sankey, graph, tree, decomposition, lineage, DuPont, or process-chain diagrams until stage/rail/node, label, gutter, and edge-bend spacing has been calculated.
- Do not accept layouts where section titles, layer/stage/lane titles, group captions, or parent labels overlap, touch, or visually attach to component cards, node cards, borders, connector lines, or child labels. Treat this as a `VIS-*` visual defect and repair spacing before handoff.
- Do not accept overlap or stacking between business-question text, conclusion text, component text, legends, labels, chart marks, table cells, cards, diagram nodes, connector lines, controls, or state messages. Treat text-text, text-graphic, and graphic-graphic collisions as `VIS-*` defects.

## Output Format

When using this skill, provide:

1. Component inventory and loaded reference files.
2. Viewport and size assumptions for each component, including parent block and sub-block viewport when relevant.
3. Positioning and alignment rules for implementation-ready components: coordinate system, main visual center, slot rectangles, x/y algorithms, local-filter geometry when present, height/horizontal budgets, size tiers, and fallback order.
4. Style decisions: typography, spacing, color, border/shadow, state, and interaction feedback.
5. Anti-AI component gate: visual cliché scan, copy specificity, state/accessibility coverage, and any approved exceptions.
6. Report decision component gate when applicable: `reportDecisionRisk`, `RPT-*` findings, metric口径 support, chart/table task fit, data story role, realistic data state, linkage/action/trust proof, and industry vocabulary check.
7. Fit decisions: label density, overflow, exact-value disclosure, scroll/zoom/fullscreen/drawer/table fallback.
8. Design reasonableness status and any `DESIGN-*` findings that affected component choice or fit.
9. Implementation notes for ECharts/S2/DOM/CSS behavior where relevant.
10. Self-check result: overlap, clipping, truncation, contrast, resize, aspect-ratio/geometry integrity, hover/focus/touch, loading/empty/error states.

## Quality Checklist

- Each component has a stable body viewport and explicit overflow policy.
- Implementation-ready component specs include `位置排布与对齐规则`: `W/H/P/CW/CH`, main visual center, slot x/y/width/height, alignment, size tiers, state geometry, and degradation order.
- Component-internal filters pass the local-filter check: `组件内筛选区 / 局部筛选区` is documented, scope is current-component only, `2-4` short options use capsule, `>4` or failed fit uses dropdown, multiple groups collapse to panel/popover/drawer, and the filter does not overlay or compress the decision-critical value/plot/table.
- Analysis & Insight components pass the decision-copy check: one card answers one point, conclusion appears before evidence, visible decision items stay `<=3`, evidence/action/trust are present or explicitly not applicable, local filters affect only this component, overflow has a tooltip/drawer/detail route, and the visual treatment stays lighter than the chart/table it explains.
- Each component passes the anti-AI style gate: style choices are tokenized and product-specific, copy is concrete, imagery/icons are meaningful, and motion is stateful rather than decorative.
- Report components pass the report decision gate: primary metrics have口径 and trust context, chart/table choices answer a named task, realistic data states are supported, and detail/action paths are not hidden by visual polish.
- Unresolved `RPT-METRIC-SHELL`, `RPT-DECORATIVE-CHART`, `RPT-VISUAL-OVER-DATA`, or `RPT-NO-ACTION` findings block component acceptance for implementation-ready report work.
- Primary component text does not use generic AI/SaaS slogans without concrete behavior or evidence.
- Each internal sub-block has a stable viewport and explicit overflow/state policy before component acceptance.
- Sub-block viewports preserve `5px` spacing from the parent body edge and `5px` spacing from sibling sub-blocks.
- Composite Panels pass the one-theme/one-primary check: the title names the shared analysis topic rather than concatenating child chart names, one child is visibly primary, child count defaults to `2-3` and stays `<=4` for normal report cards, the primary child owns `50-70%` of visual weight, all child components share the declared filter/selection context unless an exception is labeled, shared legends/units are not duplicated, and the responsive fallback hides P4/P3/P2 before shrinking P1 below its minimum size.
- No-data masks inside composite blocks pass hierarchy checks: all child sub-blocks empty uses one parent mask; partial empty uses affected sub-block masks that cover sub-block title/label/control plus component body.
- Component body titles are off by default when the surrounding block/container already owns the title; chart/table/KPI bodies do not duplicate the block/page title. KPI body metric labels are also off by default when they repeat the block title.
- Component choice is reasonable for the business task and data shape; unresolved `P0`/`P1` `DESIGN-*` findings are not styled over.
- Component size is sufficient for the selected visual type and data density; cramped/narrow/small components are enlarged, split, scrolled, zoomed, or moved to fullscreen/drawer before pass.
- Fixed-height cards/KPI tiles/navigation items pass the internal height-budget check: declared padding, explicit line-height rows, row count, vertical gaps, badge/status/footer heights, and `requiredContentHeight <= componentHeight`.
- DOM overflow checks fail when a fixed-height card/navigation item has `scrollHeight > clientHeight` or `scrollWidth > clientWidth`, even if screenshot review misses the crop.
- Standard charts mapped to ECharts use actual ECharts options/series and runtime interaction; there is no dependency-only ECharts import paired with hand-authored SVG/HTML/CSS/canvas chart marks.
- SVG/canvas/ECharts/custom graphics preserve their intended aspect ratio; curves, gauges, maps, nodes, and icons are not squeezed, stretched, or warped by the container.
- Radar charts pass the radar fit check: dimensions `5-8` preferred and `<= 10` maximum, visible series `<= 3`, standardized shared scale when units differ, circular coordinate system, reserved outer label ring, separate filter/legend/metric zones, and tooltip access to exact/raw values.
- Nested summary/KPI/tile layouts pass internal fit for long labels, values, units, helper text, and action text; no decision-critical content is clipped by nowrap/ellipsis without disclosure.
- Line, area, bar, and other category-axis charts have an explicitly sorted x-axis, and labels/points/values/tooltips are derived from the same sorted row order.
- Line/area charts pass the trend-geometry check: sparse points center/symmetrically distribute, dense points use point/label sampling or zoom/brush, y-axis baseline is declared, and missing values break rather than becoming zero.
- Target/actual bar charts pass the target-encoding and plot-budget check: target line/tick/grouped bars are chosen from the data shape; title/subtitle, metric strip, legend, y-axis band, right target-label gap, plot area, x-axis band, and footer are reserved before labels and bars render.
- Combo charts pass the relationship-fit check: bar = scale/amount/count, line = rate/trend/efficiency, target/reference = standard; shared x-axis categories align bars, line points, labels, tooltip guide, and target/reference markers; `plotH >= CH * 0.48`; bar series `<=2`, line series `<=2`, target/reference `<=2`, total visible series `<=4`; dual axes show units on both sides and expose exact values in tooltip.
- Detail Tables pass the row-evidence fit check: row grain and primary key are explicit, default sort is named, visible columns are priority-limited, primary/text columns left-align, numeric/percent columns right-align, status uses lightweight badges, operations are capped, fixed header/frozen columns are applied only when needed, `tableBodyAreaH >= CH * 0.55`, and `visibleRowCount >= 4-6` by default with tooltip/detail/export access to hidden exact values.
- Grouped Table Headers pass the header-structure fit check: column groups are business-meaningful, `columnTree` leaves map to real fields, `colSpan`/`rowSpan` are computed from visible leaves and `maxDepth`, parent widths equal child leaf-width sums, depth is `<=3` unless a documented collapse/fullscreen exception exists, the whole multi-level header is fixed during body scroll, left row dimensions/primary columns freeze during horizontal scroll, component-local filters and per-column header filters are visually and semantically separate, and the header does not compress useful body rows below budget.
- Pivot Tables pass the cross-summary fit check: row and column dimensions are explicit, measures and aggregation functions are named, percentage/rate totals recompute from numerator/denominator, subtotal/grand-total placement is declared, row headers freeze when horizontal scroll exists, column headers freeze when vertical scroll exists, `pivotAreaH >= CH * 0.55`, visible measures stay `1-3` by default and `<=5`, row/column density has scroll/virtualization fallback, and tooltip/drilldown exposes exact path and formula evidence.
- ECharts charts with visible x-axis labels plus bottom legends pass the reserved-space check: `grid.containLabel = true`, `grid.bottom >= 56px`, and legend-to-axis-label distance is visually clear.
- Small-card donut/pie charts pass the legend/label fit check: bottom legend is used by default; right legend has a width budget and is used only when the card is wide enough and outside labels are closed or key-label-only; `legendBandHeight`, `labelLineBudget`, `radius`, and `center` are declared; low-value labels may be hidden from permanent outside labels but remain available through tooltip/legend; no title/legend/label-line/label/center collision or out-of-bounds rendering exists.
- Pie/donut charts pass the composition fit check: categories `2-6` preferred, `<= 8` before merge, deterministic `Top5 + 其他` or approved merge rule, no negative values, no fake all-zero shares, center text fits the donut hole, and exact category value/percent/rank/change-rate remain available in tooltip/detail.
- KPI cards pass the value-anchor check: core value zone is centered in the body, occupies at least 40% of main visual height, and title/description placement does not create large unused blank space.
- KPI cards pass the value-group placement check: `value + unit` is measured and centered as one group, with a 4-6px unit gap and no detached unit alignment.
- Scatter/bubble charts pass the relationship-fit check: X/Y metrics and units are explicit, axes have range/baseline behavior, point count drives radius/opacity/label density, bubble size uses bounded sqrt mapping when present, reference lines/quadrants/trend lines are weak and data-driven, and tooltip/detail exposes exact object values.
- Parallel coordinates pass the multi-metric-pattern fit check: each visible line represents one object/sample, visible dimensions are `3-8` by default and `<=12` with filtering/scroll, `plotH >= CH * 0.48`, `axisGap >= 56px`, every axis declares range/unit/direction or standardized mode, ordinary sample labels are hidden, line opacity follows sample count, Top/anomaly/selected lines are highlighted above muted lines, brush filters update selected sample counts, legend and local filters are separate, and tooltip/detail exposes the full object values and scale/normalization context.
- Map/geographic charts pass the spatial-fit check: geography is the actual decision dimension, region codes or lon/lat fields are declared, the map resource/projection preserves aspect ratio, `mapAreaH >= CH * 0.55`, labels are key-only, visualMap/legend does not cover important geography, dense points use clustering/heatmap and dense flows use TopN, and tooltip/detail exposes exact region/point/flow values.
- Candlestick/K-line charts pass the OHLC-fit check: ordered `open/high/low/close` rows are valid, unit and market color convention are explicit, `mainChartH >= CH * 0.45`, volume/indicator subplots remain secondary, visible candle count drives body width and dataZoom, time labels are sampled, only high/low/latest/selected labels persist, and tooltip/detail exposes full OHLC/volume/MA values.
- Boxplot charts pass the distribution-fit check: Q1/median/Q3/IQR/whisker/outlier rule and sample counts are explicit, `plotH >= CH * 0.50`, boxes align to category centers, grouped series stay within `S <= 3` and readable category count, outliers are stable and not overplotted, only key labels persist, and tooltip/detail exposes the full five-number summary plus outlier details.
- Matrix/time/calendar/correlation heatmaps pass the density-fit check: row/column/value/aggregation contracts are explicit, `matrixH >= CH * 0.45`, color scale and visualMap are visible, missing and zero cells are distinct, cells stay within `12-44px` unless scroll/aggregation is declared, value labels appear only when cells are large enough, row/column labels are sampled before collision, and tooltip/detail exposes exact cell values.
- Gauge charts pass the status-fit check: one bounded metric is explicit, min/max and overflow behavior are declared, `gaugeAreaH >= CH * 0.50`, the center value plus unit is the strongest element, arc geometry preserves semicircle/circle proportions, target/threshold labels do not collide with ticks or center value, status colors follow business meaning rather than raw high/low, local filters stay in the title area, and tooltip/detail exposes current value, range, target, gap, threshold/status, period, source, and denominator/zero states.
- Path/user/process path charts pass the flow-fit check: start/end and step order are explicit, `pathAreaH >= CH * 0.52`, main path stays centered on the path canvas, branch nodes are ranked and bounded, node count and link count stay within path density limits or use Top N/aggregation, path labels are key-only, arrows stop at target-node edges, legend/filter controls are separate, and tooltip/detail exposes exact node and transition evidence.
- Funnel charts pass the conversion-fit check: ordered stages and shared population/cohort logic are explicit, `funnelAreaH >= CH * 0.52`, the horizontal bar or funnel body keeps left stage labels and right value/share labels in reserved columns, `3-6` stages are preferred and `>10` stages use fold/scroll/table fallback, only the maximum/key loss is permanently highlighted, legend and local filters stay separate, and tooltip/detail exposes value, entry share, stage conversion, drop value/rate, total conversion, target/comparison, period, source, and denominator-zero behavior.
- Sankey diagrams pass the flow-fit check: source/target/value links and node schema are explicit, layers flow left-to-right or in a declared direction, `sankeyAreaH >= CH * 0.55`, main flows remain visually dominant, long-tail flows use Top N/`其他`, node labels are key-only, flow labels appear only for thick/key ribbons, legend/filter controls are separate, and tooltip/detail exposes exact source, target, value, shares, conversion/loss, period, source, and aggregation evidence.
- Treemap/rectangular tree maps pass the composition-area fit check: hierarchy fields and parent/leaf totals are explicit, the area metric is non-negative and additive, `treemapAreaH >= CH * 0.55`, Top N/`其他` aggregation is applied before tiny leaves take over, parent groups remain readable, child labels obey rectangle thresholds, color metric/legend semantics are not confused with area, breadcrumbs or drilldown appear for deep levels, and tooltip/detail exposes full path, value, percent of total, percent of parent, color metric, rank, source, and aggregation rule.
- Sunburst charts pass the hierarchy-path composition fit check: hierarchy fields or `children` are explicit, the angle metric is non-negative and additive, `sunburstAreaH >= CH * 0.55`, visible levels default to `2-3` and never exceed `4` without drilldown, `ringW >= 18px`, center text fits `innerR * 1.5`, Top N/`其他` aggregation prevents tiny sectors from dominating, labels appear only when `sectorAngle >= 10deg`, `ringW >= 22px`, and arc length fits the text, color/legend semantics are distinct from filters, breadcrumb/drilldown handles deep paths, and tooltip/detail exposes full path, value, percent of total, percent of parent, color metric, rank, source, and aggregation rule.
- Tree/hierarchical tree charts pass the hierarchy-fit check: root node and parent-child fields are explicit, `treeAreaH >= CH * 0.55`, orientation matches depth/breadth, default expansion stays within `2-3` levels unless space proves more, child branches use Top N/`+N` aggregation when dense, parent nodes align to child centers, connector paths stop at node edges, labels fit inside nodes or disclose through tooltip, legend/filter controls are separate, and tooltip/detail exposes exact node, parent, children, value, status, and source evidence.
- Relation/network graphs pass the structure-fit check: node/edge schema and relationship direction are explicit, `graphH >= CH * 0.55`, node count and edge count stay within readable density or use filtering/aggregation/local exploration, core/selected/anomaly labels are prioritized, ordinary labels and edge labels hide by default, edges stay weaker than nodes, fitView keeps the graph inside the canvas, and tooltip/detail exposes exact node/edge evidence.
- Tables with more than 8 visible columns or natural grouped fields use complex/grouped headers, or the flat-header exception is documented.
- Repeated peer cards/charts/tiles use internal exact `M * N` distribution only when `actualTotal > 4`; prime `actualTotal` first becomes `layoutTotal = actualTotal + 1`, `layoutTotal = M * N`, `M >= N`, and `M - N` is minimal among valid factor pairs; sub-block viewports pass min-size checks and parent blocks pass the height-capacity check with `heightExpansionRows = ceil(N * 2 / 3)`.
- Titles, units, labels, legends, controls, pagination, and states fit without overlapping data marks.
- Business-question text, conclusion text, component labels, chart marks, diagrams, tables, and cards do not overlap, stack, or visually merge.
- Section titles, layer/stage/lane titles, and group captions have their own reserved title band and at least 16px safe spacing from the nearest card, node, connector, or label.
- Hidden or abbreviated values are inspectable.
- Dense charts use label budgets and tooltips; dense tables use S2 or horizontal scroll.
- Complex diagrams use zoom/pan, minimap, drawer, or fullscreen instead of overflowing.
- Complex diagrams keep layer numbers, stage/layer/lane titles, group captions, labels, nodes, connectors, and edges at least 16px apart and reserve rail, title-band, and edge-bend space.
- Primary filters use Element Plus or project design-system select/dropdown/date/cascader controls, or a fully styled native select only for baseline prototype acceptance.
- Controls inside components have clear selected/hover/disabled/loading states and do not resize the block.
- Hover/focus states for cards, KPI tiles, chart/table containers, and compact controls do not move, scale, or visually escape the assigned viewport; border glow or inset glow is preferred over offset animation.
- Text summaries and conclusions inherit the surrounding layout tokens and do not look like unrelated inserted bands.
- Change-rate indicators pass positive-red-up / negative-green-down semantics and use `%` in Chinese UI.
- Loading, empty, error, no-permission, and stale states preserve geometry.
- Runtime resize or visual verification is performed when the component is implemented, including at least one viewport/container size different from the design baseline for graphics that use SVG, canvas, ECharts custom paths, maps, gauges, or complex diagrams.

## Avoid

- Do not duplicate detailed reference rules in final answers; cite the loaded references and apply their decisions.
- Do not use decorative cards, gradients, or visual noise when the component's job is analytical scanning.
- Do not make components look "premium" by adding generic gradients, glass, glow, huge radii, abstract AI imagery, or ornamental animation.
- Do not accept generic empty copy in component headings, helper text, empty states, or errors.
- Do not use charts as visual variety when a table, detail drawer, decomposition, or action block is the actual decision support.
- Do not hide metric口径, source/freshness, baseline, anomaly marker, or owner/action context merely to keep a KPI/chart card cleaner.
- Do not add internal chart/table/card titles that duplicate the page or block title.
- Do not make a component look compact by making it too narrow, too small, crowded, or unreadable.
- Do not hide required table columns, action text, KPI values, or alert labels merely to make the layout look cleaner.
- Do not use a chart when the exact-value task requires a table or detail list.
