# Frontend Implementation Gates

Detailed frontend implementation rules moved out of `SKILL.md`. Load this before changing frontend code or judging frontend readiness.

## Reinforced Constraints

- Resolve `prototypeSourcePath` and `frontendTargetPath` before file edits. Treat upstream prototype source as read-only unless the user explicitly designates it as the frontend target.
- Preflight does not authorize the whole implementation without further thought. Before every non-trivial edit, renderer choice, HTML/sample conversion, data-binding, dependency addition, or readiness decision, run the action reflection loop from `$quality-gate-validation` `references/preflight-understanding-gate.md`; stop or reroute when the planned action fails constraints, source authority, or design reasonableness.
- For every frontend code file that will be changed, locate or create its sidecar code ledger before editing, read it, and use its feature inventory/range history to decide the edit. After editing, append a version entry with feature list, changed code ranges, modified content, affected props/events/API/env/filter contracts, verification, and rollback notes. Default sidecar path: `<code-file-dir>/__change_logs__/<code-file-name>.changes.md`.
- Do not implement "modern/high-end/tech" aesthetics as a substitute for product context. User task, real content, first action, data structure, state coverage, accessibility, and tokenized implementation must be known or explicitly marked as gaps before readiness.
- Do not add generic purple-blue gradients, glassmorphism, black neon, glow buttons, decorative orbs/particles/grids, oversized radius, abstract AI imagery, or ornamental motion unless the approved design source requires them and the exception is documented.
- Do not ship generic copy in primary headings, CTAs, empty/error states, or key summaries. Replace broad slogans with concrete user actions, data objects, system behavior, conditions, limits, evidence, or next steps.
- Avoid sample-like implementation: hardcoded one-off colors/spacing/radius, untyped random rows, repeated div/card markup, missing reusable components, missing ARIA/labels/focus-visible, and missing loading/empty/error/no-permission/disabled/success states.
- For report frontends, do not ship a generic KPI/chart shell. Every primary metric needs formula/denominator, grain, period, source/freshness, unit/precision, baseline, owner/action, and drilldown/detail behavior in the view model or explicit handoff gap.
- Every visible report metric needs a numeric display contract in the view model or formatter metadata: value type, raw/display unit, display scale, screen precision, tooltip/export precision, rounding mode, null/zero/denominator-zero behavior, negative-zero handling, formula precision policy, and formatter ownership.
- Use a central number formatter or project utility for governed report metrics. Do not scatter `toFixed`, `Math.round`, `parseInt`, string slicing, ad hoc `%` concatenation, or one-off locale formatting inside component templates, ECharts option builders, S2 cells, table formatters, tooltip formatters, or export handlers.
- Report mock/offline data used as implementation evidence must include realistic messy cases: variance, missing/zero/extreme values, long labels, non-default filter branches, freshness/source metadata, permission-limited states, and reconciliation between KPI/chart/table/drawer/export totals.
- Report filters, clickable chart marks, KPI cards, tables, drawers, exports, share links, and refresh actions must preserve active filter, period, permission, and snapshot/batch context. Selected-state-only UI changes are report integration defects.
- Report-designer/editor frontends must implement or explicitly gap data source selection, field binding, aggregation, calculated fields, filter/parameter configuration, permission/data scope, validation errors, preview, versioning, and publish flow. A left library + canvas + right properties shell is not ready.
- For self-developed frontend work without an authoritative existing stack or user-specified override, use `Vue 3 + TypeScript + ECharts + Element Plus + axios + AntV S2` as the default report frontend stack. Use AntV S2 for pivot tables, cross tables, frozen-header analytical tables, wide metric matrices, and dense comparison grids; use Element Plus for controls/forms/tables/pagination/dialogs; use ECharts for standard charts.
- Treat the default stack as an implementation contract, not a dependency checklist. When a component is mapped as an ECharts standard chart, it must instantiate ECharts or the existing project ECharts wrapper, build a data-driven `option`/`series` from the component view model, call the normal render/update/resize path, and keep ECharts tooltip/legend/emphasis behavior available. Do not satisfy an ECharts chart requirement by importing `echarts` while drawing the chart marks with hand-authored SVG/HTML/CSS/canvas.
- Hand-authored SVG is allowed for logos, icons, small trend arrows, decorative assets, or explicitly approved custom diagrams that ECharts cannot reasonably express. ECharts configured with `renderer: 'svg'` is still valid because ECharts owns the generated SVG; manually encoded bars, lines, pies, gauges, maps, parallel-coordinate axes/lines, sunburst arcs, treemap rectangles, Sankey ribbons, relation graphs, or axes are not valid standard chart implementation. Path/user/process path charts may use ECharts sankey/graph/custom series or a declared data-driven custom diagram, but not decorative hardcoded journey lines.
- If `prototypeSourcePath` is an HTML file or static page that contains chart-like `<svg>`, `<canvas>`, CSS bars, DOM arcs, or absolute-positioned chart marks, do not port those marks as implementation. Use them only to infer layout rhythm, labels, series names, categories, colors, or possible mock values, then rebuild standard charts through ECharts options/series. A custom SVG/canvas diagram must be declared before coding, data-driven, and outside the standard chart families.
- Run baseline install/typecheck/lint/test/build when feasible before large edits, and record pre-existing failures so integration work does not hide them.
- If npm dependency installation is blocked by domestic network access, use a temporary command-level mirror instead of changing global/project config: `npm install --registry=https://registry.npmmirror.com` or `npm install <package-name> --registry=https://registry.npmmirror.com`; if unavailable, replace the registry URL with `https://npm.aliyun.com/`, `https://mirrors.cloud.tencent.com/npm/`, `https://mirrors.ustc.edu.cn/npm/`, or `https://mirrors.tuna.tsinghua.edu.cn/npm/`. Use `npm ci --registry=<registry-url>` only when an existing lockfile should be restored exactly.
- Classify provider/source mode, env/proxy/base path, and auth/SSO behavior before replacing mocks or wiring request clients.
- Validate provider/API contracts and design adapters before changing production data paths. Missing provider, field, enum, formula, env, auth, permission, deployment, or testing facts must be visible gaps.
- Before wiring or accepting filter linkage, verify data completeness first: option data, provider/mock/business rows, required fields, default and non-default filter states, empty/no-permission states, and resolver/API branches must exist at the grain required by every affecting filter.
- Classify controls before wiring them: `perspective-switch`, `global-filter`, `local-filter`, or `drilldown-param`. A control that changes metric names, component set, component semantics, table headers, metric口径, or domain vocabulary is not an ordinary filter.
- Global filters, search, permission scope, pagination, sorting, drilldown, refresh, export, rankings, and aggregation must feed provider/API/resolver inputs. Component-internal filters may only operate on an already fetched component dataset.
- For snapshot/latest-period API groups, frontend may read `snapshotDate/latestPeriod/loadBatch/dataVersion` from filters, health, metadata, or snapshot response metadata and pass that context into metrics/trend/table/export requests. The backend remains responsible for using those params plus backend-injected permission/data scope to filter the source/precompute/cache/snapshot query. Frontend may reuse snapshot business payload arrays only when the API contract explicitly labels the payload as a reusable canonical snapshot or bounded component-group payload for those components.
- A filter that should affect a component must be wired to provider/API/resolver params, `filterFields`, `requiredFilters`, or an equivalent mapping. Do not leave it in `ignoredFilters`; selected-state-only filter changes are failed integration.
- When the target is a bundled report template or copied template project, preserve the template's native filter trigger/panel/popover/drawer and `filters[]` contract. Do not add a standalone filter toolbar, persistent filter bar, or extra filter drawer unless the user explicitly asks for template-level redesign.
- In bundled/copy templates, `filters[]` is for horizontal row-scope constraints. Put business domain, report theme, management object, subject area, or first-level perspective in nav/page/route/tab/segment/perspective state when it changes component schema or domain wording.
- Use `$report-component-style-design` for report component fit, KPI/chart/table readability, and metric display semantics. In visible Chinese report UI, rate, completion, variance-rate, YoY, MoM, and change labels display `%`, not `pt`, `p.p.`, or `percentage point`, unless the user explicitly requests that term.
- Metric display semantics must follow `$metric-number-display-contract`: raw values remain numeric for sorting/filtering/status/export, while labels/axis/tooltip/table/export formatting consume the declared contract.
- When implementing component-internal `local-filter` / `localFilters[]`, follow `$report-component-style-design` as the source of truth: the control affects only the current component/local group over already loaded bounded data, uses `2-4` short-option capsule vs `>4`/failed-fit dropdown, and must not overlay or compress the KPI value, chart plot, table header, legend, axis, or state message.
- Analysis & Insight widgets must be data-driven decision-support components, not hardcoded generic copy. Conclusion cards, insight cards, anomaly/risk explanations, attribution/recommendation cards, definition/data-quality/forecast notes, chart annotations, explanatory empty states, permission/no-result/delay notes, and task notes should use `visualType: 'text-summary'` only with `analysisInsightContract` or equivalent props: subtype/family, conclusion, evidence, affected object, comparison/change value when relevant, action/detail path, confidence/definition/source/freshness, local-filter scope, tooltip payload, and loading/insufficient/empty/error/no-permission/data-delay states.
- Detail Table widgets must be data-driven row-level components with declared row grain, primary key/object field, visible column priority, type/width/alignment metadata, default sort, search/sort/pagination/export provider scope, row detail/action payload, fixed header/frozen columns when needed, component-local filter limits, and stable loading/empty/error/no-permission states. Use Element Plus/project table for ordinary detail rows; use AntV S2 only for pivot/cross/wide-matrix/frozen analytical grids. Do not render all source fields as one dense flat table or apply global pagination/search/sort purely client-side over unbounded data.
- Pivot Table widgets must be data-driven S2-class analytical components with visualType `pivot`, row dimensions, column dimensions, measures, aggregation formulas/functions, rate numerator/denominator rules, subtotal/grand-total rules, hierarchy depth, sort behavior, frozen headers, row/column density fallback, conditional-formatting limits, tooltip/drilldown payload, local metric/display controls, and stable loading/empty/error/no-permission states. Use AntV S2/project S2-equivalent for real pivot/cross/wide matrices. Do not synthesize production pivots from unbounded raw rows in the browser or hand-roll merged HTML tables when S2 behavior is needed.
- Complex/grouped table headers must be implemented through the project table/S2 grouped-column model. Declare `columnTree` or nested columns, real leaf fields, units/definitions, computed `colSpan`/`rowSpan` or depth metadata, fixed whole-header behavior, frozen row/primary columns for horizontal scroll, component-local filter vs per-column header-filter separation, sort/filter icon limits, tooltip definitions, density fallback, and stable states. Do not fake merged headers with absolutely positioned text, decorative color bands, or detached CSS grids that do not scroll/freeze with the table body.
- Composite Panel widgets must be implemented as one cohesive component with an internal layout and shared state. Declare `compositePanelContract`, one shared topic, analysis sequence, primary child id, child roles/priorities/min sizes, default `2-3` children and normal max `4`, primary visual weight `50-70%`, `contentH >= CH * 0.60`, panel-level local filters, child-only filter exceptions, shared legend/unit behavior, linked hover/click state, detail-preview limits, responsive fallback, and parent/child states. Do not scatter unrelated widgets across the page grid or create nested card shells to mimic a Composite Panel.
- Combo widgets must be ECharts/data-driven from one shared ordered x-axis with bar series for scale/amount/count and line/target/reference series for rate/trend/efficiency/standard. Declare bar/line metric roles, units, left/right y-axis mapping, visible series limits, dual-axis rationale when present, category-density fallback, local-filter fit, and exact tooltip/detail. Do not implement Combo as hand-authored SVG/HTML/CSS bars/lines or use dual axes for unrelated metrics.
- Gauge widgets must be ECharts/data-driven from one bounded progress/status metric, with declared min/max range, current value, unit, clamp/overflow behavior, target/threshold/status semantics, business color direction, angle/radius/center geometry, center value, local-filter fit, and exact tooltip/detail. Do not implement gauges as decorative hand-authored arcs, fixed SVG needles, or dashboard variety when a KPI card, bar, line, table, or progress bar is the correct answer.
- Treemap/rectangular tree map widgets must be data-driven from hierarchy fields or `children`, with a non-negative additive area metric, optional color metric semantics, Top N/`其他`, label thresholds, breadcrumb/drilldown when deep, color legend/visualMap, and exact full-path/value/total-share/parent-share tooltip evidence. Do not implement a treemap as fixed decorative rectangles or use negative/rate/score values directly as area.
- Sunburst widgets must be data-driven from hierarchy fields or `children`, with a non-negative additive angle metric, optional color metric semantics, visible depth/ring-width budget, Top N/`其他`, sector label thresholds, center content, breadcrumb/drilldown when deep, color legend/visualMap, and exact full-path/value/total-share/parent-share tooltip evidence. Do not implement a sunburst as fixed decorative rings or use negative/rate/score values directly as sector angle.
- Parallel-coordinate widgets must be ECharts/data-driven from object/sample rows, `3-12` ordered dimension fields, per-axis unit/range/direction, independent or standardized scaling, sample-density opacity/sampling/aggregation, Top/anomaly/selected highlight semantics, optional brush state, and exact object tooltip/detail. Do not implement them as decorative hand-authored line webs.
- Funnel widgets must be data-driven from ordered stage rows with shared population/cohort logic, value/unit, entry share, stage conversion, drop value/rate, total conversion, stage-count fallback, target/comparison semantics when present, and exact stage tooltip/detail. Use ECharts `series.type: 'funnel'` for traditional funnels or ECharts horizontal `bar` series for report-default bar funnels; do not implement decorative hand-authored trapezoids or descending CSS blocks.
- Sankey widgets must be ECharts/data-driven from node/link rows with `source`/`target`/`value`, layer/stage order, metric unit, Top N/`其他` aggregation, flow-width mapping, node/link density rules, legend/filter separation, hover/click upstream-downstream highlight, and exact node/link tooltip/detail. Do not implement them as decorative hand-authored ribbons or use Sankey for simple ranking/composition/no-flow data.
- Tree/hierarchical tree widgets must be data-driven from root/parent-child fields or a declared `children` tree, with visible-depth/default-expanded rules, expand/collapse state, Top N/`+N` child aggregation, search/locate when dense, and node tooltip/detail evidence. Do not implement a tree chart as fixed decorative boxes and lines unless it is explicitly approved as a custom diagram with the same data contract.
- Classify the frontend surface as Haier/enterprise Web app, report/dashboard, or mixed before implementation. For Haier/enterprise report pages, apply `$haier-enterprise-app-ui-design-spec` as the inherited company application baseline and `$report-design-system-governance` as the report-specific baseline throughout implementation and handoff.
- Production paths cannot depend on unapproved mocks, offline providers, fake timers, generated rows, or demo-only SDKs. Any retained mock/offline source must be named, scoped, and excluded from `ready` production handoff unless explicitly accepted.
- Runtime readiness needs build/start evidence and browser QA evidence. Production-bound frontend work also needs monitoring/feedback/SLA notes through `$production-observability-feedback`.

## Workflow

1. Discover source and target paths. If prototype source is upstream evidence, copy or identify a writable frontend target before editing.
2. Inspect stack, package manager, env files, router, request utilities, stores/composables, mock/static data, and component consumers. If this is self-developed work and no existing stack is authoritative, align implementation to the default `Vue 3 + TypeScript + ECharts + Element Plus + axios + AntV S2` stack.
3. Classify the UI baseline: Haier/enterprise Web app, report/dashboard, or mixed. Load `$haier-enterprise-app-ui-design-spec` for Haier/enterprise application tokens, typography, color, spacing, base controls, states, and brand/logo rules; additionally load `$report-design-system-governance` for report/dashboard/BI/data-screen surfaces.
4. Run the anti-AI implementation gate: record product context, real content, forbidden visual defaults, copy specificity, edge-state coverage, accessibility requirements, and token/reuse strategy.
5. For report interfaces, run the report decision implementation gate: answer the five decision questions, prove metric dictionary completeness, map the metric tree/data story, identify realistic data cases, and define filter/drilldown/export/action/trust behavior.
6. Run baseline install/build/test when feasible and record pre-existing failures.
7. Run `$quality-gate-validation` when source code, API docs, provider samples, env/auth notes, requirements, design baseline, or runtime traces conflict.
8. Validate provider/API contract with `$api-contract-validation`.
9. Design adapters with `$data-transformation-adapter-design` when provider payloads differ from UI view models.
10. Verify env/proxy/base path/build/deploy behavior with `$frontend-env-deployment-verification`.
11. Use `$haier-sso-integration` for Haier account-center login or auth header behavior.
12. Before each source-code edit, run `$code-change-ledger-management`: read/create the sidecar ledger for the target file and use the helper script when available.
13. Before each non-trivial source edit, renderer selection, HTML/sample conversion, data-binding, dependency addition, or readiness claim, run the action reflection loop and revise or route when the action fails constraints or design reasonableness.
14. Classify controls and verify data completeness for filters before linkage: perspective switches, global filters, local filters, drilldown params, data source mode, option sources, row grain, required fields, default/non-default/empty/permission states, and resolver/API branches.
15. Replace or isolate mock data, wire perspective switches/filters/interactions/pagination/sorting/export/refresh to provider inputs, preserve template-native filter surfaces when applicable, and keep component view models stable.
16. After every source-code edit, append the sidecar ledger version entry before declaring the file done.
17. Use `$report-component-style-design` when KPI cards, charts, tables, summaries, metric units, Chinese `%` display, overflow, truncation, responsiveness, or visual readability are touched or affected by provider data.
18. Verify chart engine fidelity for every standard chart: dependency/import is not enough; source must contain an ECharts instance/wrapper and data-driven options, and runtime must show ECharts-owned rendering and interaction instead of hand-authored SVG chart marks.
19. Use `$performance-optimization` when data volume, first screen, API latency, chart/table rendering, or export performance matters.
20. Use `$production-observability-feedback` when production-bound delivery needs monitoring, runtime error/performance metrics, data refresh SLA visibility, analytics, alerts, or feedback closure.
21. Run `$frontend-runtime-qa-validation`, then produce `$frontend-function-description-documentation` for handoff.

## Quality Gate

- Do not edit upstream prototype source unless explicitly designated as target.
- Do not mark frontend implementation or repair `ready` when any changed scoped frontend code file lacks a sidecar code ledger, pre-change read evidence, or a post-change version entry with code ranges, modified content, affected contracts, and verification/blocker.
- Do not mark frontend readiness `ready` while unresolved `AI-CONTEXT-THIN`, `AI-FIRSTSCREEN-ONLY`, `AI-GENERIC-COPY`, missing state coverage, missing keyboard/focus support, or sample-like hardcoded styling remains.
- Do not use generic AI/SaaS decoration or generic copy as a substitute for real product content, design tokens, component states, and accessible interactions.
- Do not start a self-developed report frontend on an unspecified or ad hoc stack. Use the default stack or document the user/existing-project override before implementation.
- Do not leave production paths on unapproved mocks.
- Do not claim filter integration until data completeness has been checked first. Missing option rows, missing fact/business rows, missing provider fields, single-snapshot mock data, or missing resolver/API branches must be recorded as data gaps before binding is judged.
- Do not fetch broad data then apply global filters/pagination/sorting only in components.
- Do not replace missing metrics/trend/table APIs by deriving them from an undocumented snapshot response payload. Declared canonical/shared snapshot or bounded component-group payload reuse is valid; accidental frontend-side derivation is not.
- Do not treat version/scope display metadata as frontend-side correctness. If a filter or version changes business data, the frontend must send the stable params required by the API contract and verify the backend response reflects that query context.
- Do not treat a filter as integrated when it only changes selected UI state and affected component data remains unchanged. Classify `ignoredFilters` or missing mapping as binding gaps after data completeness is proven; classify single-snapshot mock/provider data or missing non-default branches as data gaps first.
- Do not add a new filter toolbar/bar to a template-based frontend when the template already owns filter invocation; update `filters[]`, native filter UI, and provider bindings instead.
- Do not put domain/theme/management-object/subject-area/perspective switching into ordinary template filters when it changes metric names, component set, table headers,口径, or domain wording.
- Do not claim frontend readiness when non-default perspective switching only changes numbers while titles, metric names, table headers, component set, specialty metrics, risk focus, or口径 labels remain default.
- Do not claim frontend readiness when dynamic navigation KPIs are stored in filter option metadata or when navigation percentages, overview KPIs, journey cards, and chart summaries do not reconcile to the same data chain.
- Do not claim frontend readiness when fixed-height navigation/cards/KPI tiles rely only on `auto` sizing, browser default line-height, or screenshot-only inspection; they must pass height budget and DOM overflow checks.
- Do not claim frontend readiness when visible Chinese rate/change/completion/YoY/MoM/variance-rate indicators still use `pt`, `p.p.`, or `percentage point` instead of `%`, unless the user explicitly requested that wording.
- Do not claim frontend readiness when visible governed metrics lack a numeric display contract, have ambiguous percent scale, show inconsistent decimals/units across KPI/chart/table/tooltip/export, use rounded labels for sorting/filtering/status, or rely on scattered ad hoc number formatting instead of the central formatter.
- Do not claim frontend readiness when a Haier/enterprise report page applies only report-specific rules and ignores inherited Haier application color, typography, spacing, base controls, states, brand/logo, and responsive baseline, unless the exception is explicitly scoped and accepted.
- Do not claim frontend readiness when a standard chart merely imports ECharts but renders the visual with hand-authored SVG/HTML/CSS/canvas chart marks. Rework it to ECharts option/series or document an explicit custom-diagram exception before marking ready; funnel charts require ordered stage data, shared population/cohort logic, value/unit, entry share, stage conversion, drop value/rate, total conversion, stage-density fallback, tooltip evidence, and ECharts `funnel` or data-driven horizontal `bar` rendering; path/user/process path charts require declared step/link data and transition evidence, treemap/rectangular tree maps require hierarchy data, non-negative additive area metric, Top N/`其他`, label thresholds, color semantics, and exact path/share tooltip evidence, sunburst charts require hierarchy data, non-negative additive angle metric, visible depth/ring-width budget, Top N/`其他`, sector label thresholds, center content, breadcrumb/drilldown, and exact path/total-share/parent-share tooltip evidence, tree/hierarchical tree charts require declared root/parent-child data, expand-collapse, and node evidence, and relation/network graphs use ECharts `graph` by default.
- Do not claim readiness for an HTML-restored or HTML-derived prototype when chart-like SVG/canvas/DOM marks were copied from the source into standard chart components. Rebuild through ECharts/data-driven view models or record a custom-diagram exception before implementation.
- Do not mark report frontend readiness `ready` while unresolved `RPT-METRIC-SHELL`, `RPT-NO-DATA-STORY`, `RPT-STATIC-FILTERS`, `RPT-NO-ACTION`, or `RPT-DESIGNER-SHELL` remains.
- Do not replace missing report metric systems with generic `Revenue/Users/Growth/Conversion` labels, decorative charts, or all-upward mock data.
- Do not claim report filter/drilldown/export integration when active context is lost between KPI, chart, table, drawer, export, refresh, and share behavior.
- Do not mark production handoff `ready` without provider/source mode, backend/API base, env/auth behavior, runtime QA evidence, retained mock status, and testing/observability handoff.
- Do not claim handoff readiness without build/runtime evidence or a precise blocker.
