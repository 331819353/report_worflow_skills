# 状态总览报表设计

## 核心定位

把状态总览型报表当作一种通用报表方法，而不是某一个经营驾驶舱模板。它的价值是让用户快速判断一个对象当前是否健康、是否达标、是否偏离、是否有风险，以及下一步应该关注哪里。

适用对象可以是企业、区域、组织、产品、项目、门店、客户、库存、服务、流程、预算、专项工作或任何需要做健康判断的管理对象。

状态总览大屏或驾驶舱可以由多个设计部分共同支撑，但必须先选一个主子类型作为首屏叙事主线，再把其他子类型作为局部辅助模块或下钻入口，避免把一页做成十类总览的拼盘。

## 通用输入

设计前先明确这些输入，缺失时要先提出假设：

- 观看者：高层、经营负责人、区域负责人、职能负责人、项目负责人、一线管理者。
- 管理对象：被判断状态的对象和层级。
- 状态总览子类型：全局经营、业务域、目标计划、流程链路、资源能力、对象资产、项目事项、区域空间、风险健康、系统服务之一；复杂大屏可再声明辅助子类型。
- 时间范围：日、周、月、季、年、项目阶段、经营周期。
- 判断基准：目标、预算、计划、阈值、时间进度、去年同期、上期、历史均值、行业/内部标杆。
- 核心指标：结果、完成、偏差、趋势、结构、风险、效率。
- 后续动作：查看诊断、查看明细、核对数据、处理异常、发起任务、进入复盘。

## 思路逻辑

从“如何判断状态”出发，而不是从“摆哪些图表”出发。

1. 定义判断问题：用户需要在 10 秒内判断什么。
2. 定义状态对象：判断的是整体、组织、项目、流程、资产还是人群。
3. 选择一个主 `statusOverviewSubtype`，必要时声明辅助子类型和场景标签。
4. 定义基准：所有关键指标都要有可比较对象。
5. 构建指标层级：结果指标说明现状，完成指标说明达标，偏差指标说明差距，趋势指标说明变化，结构指标说明对象，风险指标说明优先级。
6. 组织信息路径：先整体状态，再目标偏差，再趋势结构，最后给出风险入口。
7. 设定交互边界：总览页负责发现状态和定位入口，不承担完整根因分析、明细核对或任务闭环。
8. 输出管理结论：用短结论说明整体状态、主要偏差和优先风险。
   结论表达不等于必须新增独立横向条带；当页面来自样板/HTML 复刻时，结论必须融入原有标题区、全景图头部、分区头部、摘要卡或同级容器。

## 落地路径

按下面顺序把方法落到页面或原型：

1. 抽象业务问题
   将需求改写成一句话：在什么周期内，谁要判断哪个对象是否健康，依据什么基准，下一步要进入哪里。

2. 建立指标模型
   至少包含实际值、基准值、完成率/偏差、趋势、结构拆分和风险对象。避免只有实际值没有目标或阈值。

3. 定义状态规则
   明确红黄绿、达标/未达标、健康/预警/异常的计算口径、阈值来源和例外规则。

4. 设计信息架构
   常用层级是“整体状态 -> 目标偏差 -> 趋势变化 -> 结构分布 -> 风险清单 -> 跳转入口”。小型报表可以合并层级，但不要打乱判断顺序。

5. 映射组件
   用 KPI 卡、目标达成条、子弹图、对比柱、偏差条、趋势线、结构图、热力表、Top/Bottom、风险清单承载不同判断任务。

6. 设计交互
   支持时间、组织、业务线、对象类型、基准切换；支持 KPI 聚焦、结构联动、趋势点查看、风险对象抽屉、定义口径弹层和跨报表跳转。

7. 明确数据契约
   为每个组件定义字段、口径、基准来源、更新时间、过滤参数、下钻参数、状态规则和跳转参数。

8. 验收表达
   验收时检查用户是否能快速回答：当前好不好、差在哪里、风险对象是谁、该看哪张后续报表。

## 组件选择

- KPI 卡：承载整体健康判断和关键结果。
- 子弹图/进度条：承载实际、目标、完成率、区间状态。
- 对比柱/偏差条：承载不同对象的达标差距。
- 折线/面积图：承载趋势、拐点和连续异常。
- 热力表/矩阵/地图：承载组织、区域、对象健康分布。
- Top/Bottom：承载优秀对象和拖累对象。
- 风险清单：承载可行动对象、负责人、严重度、持续时间和下一入口。

图表选择必须服务判断任务，不为了丰富视觉而堆图。

流程、路径、流向、价值链、血缘或传导类图形不是状态总览的默认必选组件。只有当核心问题明确要求价值链、依赖关系、血缘、传导、流程转化或流向归因时，才把它作为业务必需组件；否则最多作为样板结构保留或增强项。

变化率、同比、环比、完成率差异等指标在中文页面中使用 `%` 表达。变化率类视觉语义按中国业务习惯处理：正值使用红色字体加上升 SVG/图标，负值使用绿色字体加下降 SVG/图标，零值使用中性样式。

## 独立交付与后续接口

- 本 skill 自行输出业务问题、内容块建议、关键字段、筛选器、交互入口和验收口径，保证不依赖其它 skill 也能形成可评审的报表设计。
- 页面结构、组件选择和视觉约束在本 skill 中给出最低可落地建议；需要更细的原型布局时，把这些结果作为后续输入。
- 后续衔接仅作为交付接口：进入原型或实现时，可将本 skill 的输出交给原型、布局或组件流程继续细化，本 skill 不以它们为前置依赖。
## 独立调用最低落地门禁

最终输出至少补齐：

- 数据契约：指标字段、行粒度、基准/阈值、公式、更新时间、数据源或接口依赖。
- 筛选与交互：主筛选、影响组件、下钻/抽屉/跳转参数、返回和重置行为。
- 权限与状态：组织/对象权限、导出权限、空态、加载、错误、无权限、筛选后失效状态。
- 校验方式：首屏结论、指标与明细/风险对象是否可追溯，筛选后 KPI、图表、列表是否一致。
- 后续接口：如进入原型或实现，输出当前 skill 的字段、筛选、交互、组件建议、验收口径和未决问题；后续流程可消费这些信息，但不是本 skill 的前置条件。

## 输出格式

当用户要求设计状态总览型报表时，按这个结构输出：

1. 设计定位：说明受众、对象、周期、核心判断问题。
2. 子类型与标签：声明主 `statusOverviewSubtype`、辅助子类型、场景标签、管理层级、时间节奏和表达方式。
3. 思路逻辑：说明从状态判断到偏差定位再到风险入口的路径。
4. 指标体系：列出结果、完成、偏差、趋势、结构、风险、效率指标和基准。
5. 布局框架：说明整体状态、目标偏差、趋势结构、风险入口等层级。
6. 组件映射：说明每个组件回答什么问题。
7. 交互路径：说明筛选、联动、下钻、弹层、抽屉和跳转边界。
8. 数据落地：说明字段、口径、阈值、更新时间和接口/数据源依赖。
9. 管理结论：给出整体状态、主要偏差、优先风险和下一步入口。
10. 校验清单：列出缺口、假设、风险和需要业务确认的规则。

## 质量清单

- 是否先定义管理判断，而不是先定义图表。
- 是否声明一个主状态总览子类型，并把其他子类型限制为辅助模块或下钻入口。
- 每个关键指标是否都有基准或阈值。
- 首屏是否能快速判断健康度和目标完成。
- 偏差和风险是否能定位到可行动对象。
- 结构拆分是否对应真实管理责任。
- 交互边界是否清楚，避免把总览做成无限分析页。
- 状态颜色是否有明确规则。
- 结论是否说明整体状态、主要偏差和下一步关注点。
- 样板/HTML 复刻时，新增结论是否嵌入原有等价容器，而不是另起独立条带。
- 流程/流向/传导图是否确实由核心问题驱动，而不是因为样板中出现就变成必需组件。

## 避免

- 不要把某个行业的指标硬编码成通用规则。
- 不要把状态总览做成单一驾驶舱模板。
- 不要只有实际值，没有目标、预算、计划、阈值或历史对比。
- 不要在总览页展开完整根因分析。
- 不要让风险对象藏在装饰性图表后面。
