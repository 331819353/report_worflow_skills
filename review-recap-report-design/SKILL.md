# 汇报复盘报表设计

## 核心定位

把汇报复盘类报表当作一种“把数据组织成管理叙事并沉淀行动”的通用方法。它的价值不是让用户自由探索，而是让会议参与者快速理解一段时间内发生了什么、结果如何、为什么如此、还有哪些风险、下一步由谁做什么。

适用场景可以是经营月报、季度复盘、年度总结、经营会、专题复盘、项目复盘、区域汇报、部门汇报、PMO 汇报、董事会材料或领导简报。

## 通用输入

设计前先明确这些输入，缺失时要先提出假设：

- 汇报场景：会议、汇报、复盘、简报、材料沉淀还是行动跟踪。
- 受众：高层、业务负责人、职能负责人、项目委员会、区域负责人。
- 周期和范围：月、季、年、项目阶段、专项周期、组织范围、业务范围。
- 会议目的：通报结果、做决策、对齐问题、推动资源、督办行动。
- 证据来源：状态总览、分析诊断、绩效评估、异常监控、明细、会议纪要。
- 输出形式：页面、PDF、PPT、分享链接、演示模式、版本留存。

## 思路逻辑

从“如何讲清管理故事”出发，而不是从“如何展示所有数据”出发。

1. 定义汇报目的：明确会议背景、受众、周期、主题和需要达成的管理动作。
2. 先写结论：首屏说明总体判断、主要成绩、主要差距、关键风险和下一步。
3. 说明阶段结果：展示周期内完成、未完成、改善、下滑或延迟的核心结果。
4. 抽取关键变化：对比目标、计划、预算、同期、上期、里程碑或标杆。
5. 摘要原因：引用诊断结论，但不把复盘页做成完整探索分析页。
6. 拆分问题与风险：问题是已发生的差距，风险是未来可能影响目标的因素。
7. 明确行动计划：每个重要问题或风险都要有动作、责任人、期限、预期结果和跟踪方式。
8. 支持证据和材料：能展开证据，也能导出、分享、演示和保留版本。

## 落地路径

按下面顺序把复盘方法落到页面或原型：

1. 定义复盘主题
   将需求改写成“在某周期内，围绕某主题，向某类受众说明结果、原因、风险和行动”。

2. 写出首屏结论
   先写文字结论，再决定图表。结论至少包含结果判断、主要原因、优先风险和下一步重点。

3. 选择阶段结果
   选择能代表周期表现的指标，例如收入、利润、费用、回款、项目完成、客户增长、运营质量或领域指标。

4. 组织叙事结构
   常用结构是“结论摘要 -> 阶段结果 -> 关键变化 -> 原因摘要 -> 问题风险 -> 行动计划”。

5. 映射组件
   用结论卡、KPI、目标对比、趋势、时间线、瀑布、结构图、问题清单、风险清单、行动计划表承载故事。

6. 设计证据入口
   结论可以展开证据，指标可以打开说明，问题可以查看影响，行动可以查看进度和责任。

7. 设计材料输出
   根据场景支持 PDF/PPT、分享链接、演示模式、批注、会议纪要、版本对比和历史复盘。

8. 明确数据契约
   为每个章节定义指标字段、比较基准、事件说明、证据来源、行动字段、责任人和版本号。

9. 验收表达
   验收时检查页面是否能直接用于会议阅读，并能沉淀出明确行动。

## 内容模型

- 结论摘要：总体判断、主要成绩、主要差距、重大风险、下一步重点。
- 阶段结果：完成、未完成、改善、下滑、延迟、超预算、超计划。
- 目标对比：目标、预算、计划、里程碑、上期、同期、标杆。
- 关键变化：指标变化、结构变化、贡献变化、区域变化、费用变化、项目变化、客户变化。
- 重大事件：活动、政策、市场变化、项目事件、事故、组织变化、领导决策。
- 原因摘要：用短说明承接分析诊断结果。
- 问题风险：问题、风险、影响、严重度、责任对象和可能后果。
- 行动计划：动作、责任人、期限、预期结果、跟踪指标、状态。

## 组件选择

- 结论卡/摘要块：承载首屏管理判断。
- KPI/目标对比/子弹图：承载阶段结果和目标完成。
- 趋势图/时间线：承载周期过程和重要事件。
- 瀑布图/分解图：承载原因摘要。
- 结构图/贡献图：承载关键变化。
- 问题清单/风险卡：承载需要关注的事项。
- 任务表/路线图/Gantt：承载行动计划。
- 批注/脚注/证据抽屉：承载会议说明和材料化支撑。

会议可读性优先于探索自由度。

## 行动闭环

每个重要问题或风险都应转为行动计划：

- 动作：具体要做什么。
- 责任人：谁负责，谁协同。
- 期限：什么时候完成。
- 预期结果：完成后什么指标或状态会变化。
- 跟踪方式：任务、周会、月度复盘、专项督办。
- 当前状态：未开始、进行中、阻塞、完成、延期。
- 证据：诊断、明细、附件、会议纪要、任务记录。

没有行动闭环的复盘，只是回顾材料。

## 与其他报表的边界

- 只判断当前健康度，用 `status-overview-report-design`。
- 要展开完整原因，用 `analysis-diagnostic-report-design`。
- 要查看具体记录，用 `detail-query-report-design`。
- 要持续监控预警，用 `anomaly-monitoring-report-design`。
- 要推进会议行动，用 `operational-execution-report-design`。

## 独立调用最低落地门禁

当本 skill 未经 `report-design-workflow` 直接使用时，不要停留在方法论。最终输出至少补齐：

- 数据契约：指标字段、行粒度、基准/阈值、公式、更新时间、数据源或接口依赖。
- 筛选与交互：主筛选、影响组件、下钻/抽屉/跳转参数、返回和重置行为。
- 权限与状态：组织/对象权限、导出权限、空态、加载、错误、无权限、筛选后失效状态。
- 校验方式：结论、章节、证据、行动计划、导出/PPT/PDF、版本留存是否能形成可复盘闭环。
- 后续契约：如果要进入原型或实现，必须交给 `report-info-component-mapping` 生成绑定矩阵。

## 输出格式

当用户要求设计汇报复盘类报表时，按这个结构输出：

1. 设计定位：说明复盘场景、受众、周期、主题和会议目的。
2. 叙事主线：说明从结果到原因、风险、行动的故事路径。
3. 内容体系：说明阶段结果、目标对比、关键变化、事件、问题、风险和行动。
4. 布局框架：说明结论摘要、结果、变化、原因、风险、行动层级。
5. 组件映射：说明每个组件承担的叙事角色。
6. 交互路径：说明章节导航、证据展开、批注、抽屉、跳转、导出和版本。
7. 行动闭环：说明责任人、期限、预期结果、跟踪方式和状态。
8. 数据落地：说明字段、比较基准、事件、证据、版本和接口依赖。
9. 汇报结论：用会议可读的方式总结结果、原因、风险和下一步。

## 质量清单

- 是否结论先行，而不是图表先行。
- 是否解释了发生什么、结果如何、为什么、风险是什么、下一步做什么。
- 指标是否服务叙事，而不是分散注意力。
- 原因是否是摘要层级，并保留诊断入口。
- 问题和风险是否有优先级。
- 重要问题是否都有行动、责任人、期限和预期结果。
- 是否考虑 PDF/PPT、分享、批注、演示和版本留存。
- 最终结论是否适合会议直接阅读。

## 避免

- 不要把复盘做成没有叙事的仪表盘。
- 不要把结论藏在大量图表之后。
- 不要在复盘页重复完整诊断过程。
- 不要列风险却没有行动。
- 不要生成没有责任人和期限的行动项。
## Execution Completeness Gate

Before finalizing work with this skill, verify the following items explicitly:

1. Scope and trigger reliability: confirm the request truly matches this skill. General report-design skills must stay independent of workflow function words such as `原型设计`, `技术方案`, `前端开发`, `后端开发`, or `测试`; workflow-specific skills may use those words only when they are part of the actual phase intent.
2. Input condition handling: classify inputs as complete, partial, missing, conflicting, or runtime-only. Continue with a minimal useful artifact when safe, but mark assumptions, blockers, owners, and confirmation questions instead of inventing source fields, formulas, permissions, URLs, credentials, or business rules.
3. Flow completeness and feasibility: execute the workflow in order, split broad requests into smaller stages, and validate that each stage has the artifacts needed by the next stage before producing final output.
4. Constraint enforcement: apply the hard constraints, reference-loading rules, technology boundaries, security rules, and avoid-lists in this skill and its referenced files.
5. Output completeness: include the core deliverable, key decisions, data/source or evidence trace, missing-information list, self-check result, and next-step handoff details required by the user scenario.
6. Self-check before response: review process completeness, logical feasibility, missing-input coverage, decomposition, constraints, output integrity, generality, and trigger hygiene; repair any gap found before delivering.
