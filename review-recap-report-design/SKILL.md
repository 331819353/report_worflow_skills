# 汇报复盘报表设计

## 核心定位

把汇报复盘类报表当作一种“把数据组织成管理叙事并沉淀行动”的通用方法。它的价值不是让用户自由探索，而是让会议参与者快速理解一段时间内发生了什么、结果如何、为什么如此、还有哪些风险、下一步由谁做什么。

适用场景可以是经营月报、季度复盘、年度总结、经营会、专题复盘、项目复盘、区域汇报、部门汇报、PMO 汇报、董事会材料或领导简报。

汇报复盘可以包含多部分设计思路，设计时先按“汇报复盘对象 + 复盘主题 + 管理用途”选择一个主要 `reviewRecapSubtype`。

汇报复盘类不是数据看板，而是“有结论的数据叙事”。默认叙事链为：

```text
背景目标 -> 执行过程 -> 结果达成 -> 差异变化 -> 原因解释 -> 经验教训 -> 风险问题 -> 后续行动
```

## 通用输入

设计前先明确这些输入，缺失时要先提出假设：

- 汇报场景：会议、汇报、复盘、简报、材料沉淀还是行动跟踪。
- 汇报复盘子类型：从周期经营、目标计划/OKR、项目任务交付、策略活动、产品业务专题、客户市场渠道、流程运营履约、异常事故问题、风险合规整改、资源预算投入产出、组织协同团队管理、决策跟踪行动闭环、标杆案例经验沉淀中选择一个主类型。
- 受众：高层、业务负责人、职能负责人、项目委员会、区域负责人。
- 周期和范围：月、季、年、项目阶段、专项周期、组织范围、业务范围。
- 管理用途：信息同步、管理决策、阶段总结、经验沉淀、责任复盘、资源申请、行动闭环。
- 原始目标：目标、计划、预算、里程碑、关键假设、资源投入。
- 实际结果：完成情况、目标差距、关键变化、超预期/未达预期事项。
- 证据来源：状态总览、分析诊断、绩效评估、异常监控、明细、会议纪要。
- 上期承接：上期行动项、负责人、完成状态、延期原因、关闭结果。
- 输出形式：页面、PDF、PPT、分享链接、演示模式、版本留存。

## 思路逻辑

从“如何讲清管理故事”出发，而不是从“如何展示所有数据”出发。

1. 判定 `reviewRecapSubtype`：按汇报复盘对象、复盘主题和管理用途分类，且只选一个主类型。
2. 定义汇报目的：明确会议背景、受众、周期、主题和需要达成的管理动作。
3. 先写结论：首屏说明总体判断、主要成绩、主要差距、关键风险和下一步。
4. 还原目标和计划：说明原始目标、关键假设、投入资源、里程碑或管理要求。
5. 说明阶段结果：展示周期内完成、未完成、改善、下滑或延迟的核心结果。
6. 抽取关键变化：对比目标、计划、预算、同期、上期、里程碑或标杆。
7. 摘要原因：引用诊断结论，但不把复盘页做成完整探索分析页。
8. 沉淀经验教训：说明哪些做法可复用，哪些失误要避免，以及适用边界。
9. 拆分问题与风险：问题是已发生的差距，风险是未来可能影响目标的因素。
10. 明确行动计划：每个重要问题或风险都要有动作、责任人、期限、预期结果和跟踪方式。
11. 承接上期行动：检查上次复盘或会议行动项是否完成，未闭环事项必须进入本次行动区。
12. 支持证据和材料：能展开证据，也能导出、分享、演示和保留版本。

## 落地路径

按下面顺序把复盘方法落到页面或原型：

1. 定义复盘主题
   写清 `reviewRecapSubtype`，并将需求改写成“在某周期内，围绕某主题，向某类受众说明背景目标、结果、原因、风险、经验和行动”。

2. 写出首屏结论
   先写文字结论，再决定图表。结论至少包含结果判断、主要原因、优先风险和下一步重点。

3. 选择阶段结果
   选择能代表周期表现的指标，例如收入、利润、费用、回款、项目完成、客户增长、运营质量或领域指标。

4. 组织叙事结构
   常用结构是“结论摘要 -> 阶段结果 -> 关键变化 -> 原因摘要 -> 问题风险 -> 行动计划”。

5. 映射组件
   用结论卡、KPI、目标对比、趋势、时间线、瀑布、结构图、问题清单、风险清单、行动计划表承载故事。

6. 设计证据入口
   结论可以展开证据，指标可以打开说明，问题可以查看影响，行动可以查看进度和责任。

7. 设计材料输出
   根据场景支持 PDF/PPT、分享链接、演示模式、批注、会议纪要、版本对比和历史复盘。

8. 明确数据契约
   为每个章节定义指标字段、比较基准、事件说明、证据来源、行动字段、责任人、上期行动承接和版本号。

9. 验收表达
   验收时检查页面是否能直接用于会议阅读，并能沉淀出明确行动；若有上期行动项，必须能解释完成、延期、关闭或升级结果。

## 内容模型

- 结论摘要：总体判断、主要成绩、主要差距、重大风险、下一步重点。
- 阶段结果：完成、未完成、改善、下滑、延迟、超预算、超计划。
- 目标对比：目标、预算、计划、里程碑、上期、同期、标杆。
- 关键变化：指标变化、结构变化、贡献变化、区域变化、费用变化、项目变化、客户变化。
- 重大事件：活动、政策、市场变化、项目事件、事故、组织变化、领导决策。
- 原因摘要：用短说明承接分析诊断结果。
- 问题风险：问题、风险、影响、严重度、责任对象和可能后果。
- 行动计划：动作、责任人、期限、预期结果、跟踪指标、状态。

## 组件选择

- 结论卡/摘要块：承载首屏管理判断。
- KPI/目标对比/子弹图：承载阶段结果和目标完成。
- 趋势图/时间线：承载周期过程和重要事件。
- 瀑布图/分解图：承载原因摘要。
- 结构图/贡献图：承载关键变化。
- 问题清单/风险卡：承载需要关注的事项。
- 任务表/路线图/Gantt：承载行动计划。
- 批注/脚注/证据抽屉：承载会议说明和材料化支撑。

会议可读性优先于探索自由度。

## 行动闭环

每个重要问题或风险都应转为行动计划：

- 动作：具体要做什么。
- 责任人：谁负责，谁协同。
- 期限：什么时候完成。
- 预期结果：完成后什么指标或状态会变化。
- 跟踪方式：任务、周会、月度复盘、专项督办。
- 当前状态：未开始、进行中、阻塞、完成、延期。
- 证据：诊断、明细、附件、会议纪要、任务记录。

没有行动闭环的复盘，只是回顾材料。

## 与其他报表的边界

- 只判断当前健康度，用 `status-overview-report-design`。
- 要展开完整原因，用 `analysis-diagnostic-report-design`。
- 要查看具体记录，用 `detail-query-report-design`。
- 要持续监控预警，用 `anomaly-monitoring-report-design`。
- 要推进会议行动，用 `operational-execution-report-design`。

## 独立交付与后续接口

- 本 skill 自行输出业务问题、内容块建议、关键字段、筛选器、交互入口和验收口径，保证不依赖其它 skill 也能形成可评审的报表设计。
- 页面结构、组件选择和视觉约束在本 skill 中给出最低可落地建议；需要更细的原型布局时，把这些结果作为后续输入。
- 后续衔接仅作为交付接口：进入原型或实现时，可将本 skill 的输出交给原型、布局或组件流程继续细化，本 skill 不以它们为前置依赖。
## 独立调用最低落地门禁

最终输出至少补齐：

- 数据契约：指标字段、行粒度、基准/阈值、公式、更新时间、数据源或接口依赖。
- 子类型门禁：必须声明唯一 `reviewRecapSubtype`、汇报复盘对象、复盘主题和管理用途。
- 叙事门禁：必须先给结论，并还原背景目标、执行过程、结果达成、关键变化、原因摘要、经验教训、风险问题和后续行动。
- 行动闭环门禁：重要问题和风险必须有动作、责任人、截止时间、预期结果、跟踪指标和当前状态；如有上期行动项，必须说明完成/延期/关闭/升级。
- 筛选与交互：主筛选、影响组件、下钻/抽屉/跳转参数、返回和重置行为。
- 权限与状态：组织/对象权限、导出权限、空态、加载、错误、无权限、筛选后失效状态。
- 校验方式：结论、章节、证据、行动计划、导出/PPT/PDF、版本留存是否能形成可复盘闭环。
- 后续接口：如进入原型或实现，输出当前 skill 的字段、筛选、交互、组件建议、验收口径和未决问题；后续流程可消费这些信息，但不是本 skill 的前置条件。

## 输出格式

当用户要求设计汇报复盘类报表时，按这个结构输出：

1. 设计定位：说明复盘场景、受众、周期、主题和会议目的。
2. 子类型判断：声明唯一 `reviewRecapSubtype`，并说明为什么不是其他子类型。
3. 叙事主线：说明从背景目标、执行过程、结果、变化、原因、经验、风险到行动的故事路径。
4. 内容体系：说明阶段结果、目标对比、关键变化、事件、问题、风险、经验和行动。
5. 布局框架：说明结论摘要、结果、变化、原因、经验、风险、行动层级。
6. 组件映射：说明每个组件承担的叙事角色。
7. 交互路径：说明章节导航、证据展开、批注、抽屉、跳转、导出和版本。
8. 行动闭环：说明上期行动承接、本期行动、责任人、期限、预期结果、跟踪方式和状态。
9. 数据落地：说明字段、比较基准、事件、证据、版本和接口依赖。
10. 汇报结论：用会议可读的方式总结结果、原因、风险、经验和下一步。

## 质量清单

- 是否结论先行，而不是图表先行。
- 是否选择了唯一 `reviewRecapSubtype`。
- 是否还原原始目标、计划、关键假设和执行过程。
- 是否解释了发生什么、结果如何、为什么、风险是什么、下一步做什么。
- 是否区分事实、判断、推断和建议。
- 指标是否服务叙事，而不是分散注意力。
- 原因是否是摘要层级，并保留诊断入口。
- 是否沉淀经验教训及适用边界。
- 问题和风险是否有优先级。
- 重要问题是否都有行动、责任人、期限和预期结果。
- 是否承接上期行动项，并说明完成、延期、关闭或升级。
- 是否考虑 PDF/PPT、分享、批注、演示和版本留存。
- 最终结论是否适合会议直接阅读。

## 避免

- 不要把复盘做成没有叙事的仪表盘。
- 不要把结论藏在大量图表之后。
- 不要在复盘页重复完整诊断过程。
- 不要列风险却没有行动。
- 不要生成没有责任人和期限的行动项。
- 不要每次只提出新行动而不检查上期行动闭环。
