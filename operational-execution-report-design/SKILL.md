# 运营执行报表设计

## 核心定位

把运营执行类报表当作一种“把问题转成任务，并验证任务是否真正解决问题”的通用方法。它的价值不是被动展示数据，而是明确接下来做什么、谁负责、何时完成、当前进度、卡点风险、是否验收、业务问题是否真正关闭。

适用场景可以是整改跟踪、区域改善、回款催收、项目交付、门店巡检整改、费用压降、客户投诉处理、异常处置、专项行动、PMO 监督、领导督办或会议行动项。

运营执行可以包含多部分设计思路，设计时先按“执行对象 + 执行动作 + 闭环机制”选择一个主要 `operationalExecutionSubtype`。

运营执行类的本质是把数据洞察变成业务动作。默认闭环链路为：

```text
任务生成 -> 优先级排序 -> 责任派发 -> 上下文展示 -> 执行动作 -> 协同升级 -> 状态流转 -> 结果记录 -> 效果验证
```

## 通用输入

设计前先明确这些输入，缺失时要先提出假设：

- 执行来源：状态总览、分析诊断、绩效差距、异常告警、复盘会议、人工督办、业务流程。
- 执行子类型：从待办工作台/任务队列、线索客户/商机跟进、订单履约/交付、工单服务/售后、异常风险处置、流程审批/业务办理、项目任务/计划推进、活动战役/运营动作、资源补给/调度、财务收付/结算催办、质量整改/合规整改、巡检检查/督导、商品内容/配置发布、系统数据/运维作业、决策行动项/会议任务闭环中选择一个主类型。
- 执行目标：要解决的业务问题和预期结果。
- 任务对象：任务包、任务、子任务、节点、里程碑或处理单。
- 记录粒度：一行代表一个任务、一个业务对象、一个流程节点、一个调度事项、一个整改问题还是一个行动项。
- 执行动作：联系、审核、补货、整改、关闭、转派、催办、上传证据、处理异常、推进节点或完成办理。
- 责任体系：负责人、协同人、监督人、验收人、审批人。
- 时间规则：开始时间、截止时间、节点计划、SLA、延期规则。
- 状态规则：未开始、已分派、进行中、阻塞、延期、已提交、已验收、驳回、关闭、复开。
- 上下文信息：来源原因、客户/订单/工单/项目背景、异常原因、历史记录、依赖方、处理依据。
- 验收标准：证据、结果指标、业务状态、复发情况、审批意见。
- 效果验证：回款、转化、风险解除、异常关闭、复发率、客户确认、项目回正、指标改善。

## 思路逻辑

从“下一步必须发生什么行动”出发，而不是从“已经发生了什么结果”出发。

1. 判定 `operationalExecutionSubtype`：按执行对象、执行动作和闭环机制分类，且只选一个主类型。
2. 识别来源：任务来自问题、异常、诊断、绩效差距、会议行动、流程节点、定期规则或人工督办。
3. 定义目标：说明任务要改变的业务状态或指标。
4. 定义记录粒度：说明一行是一个任务、业务对象、流程节点、调度事项、整改问题还是行动项。
5. 建立任务模型：定义任务、子任务、责任、期限、优先级、状态、进度、证据和验收。
6. 建立动作模型：说明用户看到任务后要做什么，能否直接处理，是否需要跳转业务系统。
7. 建立进度模型：说明当前节点、计划/实际时间、逾期、阻塞、下一步和协同方。
8. 设计处理流程：支持分派、更新、转派、催办、延期、提交、验收、驳回、关闭、复开。
9. 设计结果验证：任务完成后验证指标、异常、项目、回款、成本、投诉或质量是否改善。
10. 保留可追溯记录：日志、评论、附件、审批、验收和来源链接必须可查。

## 落地路径

按下面顺序把执行方法落到页面或原型：

1. 定义执行场景
   写清 `operationalExecutionSubtype`、执行对象、来源问题、业务目标、参与角色和管理节奏。

2. 设计任务模型
   定义任务 ID、名称、来源、记录粒度、相关对象、相关指标、责任人、协同人、优先级、期限、状态、进度、卡点、证据、验收规则和结果。

3. 设计状态流
   根据业务选择“未开始 -> 已分派 -> 进行中 -> 阻塞/延期 -> 已提交 -> 已验收/驳回 -> 关闭/复开”等状态，并定义状态转换权限。

4. 设计信息架构
   常用层级是“目标概览 -> 重点任务 -> 分布责任 -> 进度跟踪 -> 风险卡点 -> 结果验证 -> 证据闭环”。

5. 映射组件
   用任务卡、任务表、看板、甘特图、时间线、步骤条、逾期清单、阻塞清单、验收表和效果对比卡承载执行任务。

6. 设计交互
   支持任务下钻、状态切换、更新进度、上传证据、转派协同、催办、延期申请、提交完成、验收关闭、驳回复工和跳转来源。

7. 明确数据契约
   为任务、节点、负责人、状态、期限、进度、上下文、处理表单、证据、日志、验收、通知、关闭标准和关联指标定义字段和接口。

8. 设计结果验证
   区分“任务提交完成”和“业务问题解决”。完成后必须能看到关联异常是否关闭、指标是否改善、项目是否回正或投诉是否复发。

9. 验收表达
   验收时检查用户是否能找到逾期、阻塞、高优先级任务，完成必要动作，并推动到验收和效果验证。

## 任务模型

- 任务身份：任务 ID、任务名称、任务类型、任务包、父任务、子任务。
- 来源：异常、诊断、绩效差距、复盘行动、人工督办、项目里程碑、客户投诉、巡检问题。
- 业务关联：指标、组织、项目、客户、合同、门店、订单、费用、付款、异常。
- 责任体系：负责人、协同人、监督人、验收人、审批人。
- 时间规则：开始、截止、计划节点、实际完成、逾期天数、SLA。
- 优先级：高/中/低、领导关注、高风险、高影响、紧急。
- 状态：未开始、进行中、阻塞、延期、已提交、已验收、驳回、关闭、复开。
- 进度：百分比、当前节点、下一步、卡点、最新更新、风险等级。
- 证据：说明、附件、截图、单据、源记录、审批、验收证明。
- 结果：完成结论、指标变化、异常关闭、问题解决、复发情况、验收意见。
- 日志：分派、转派、催办、评论、更新、提交、验收、驳回、关闭。

## 组件选择

- 看板：承载状态流和任务推进。
- 任务卡：承载优先级、负责人、期限、SLA 和快速操作。
- 任务表：承载密集督办、筛选、排序、批量操作和导出。
- 进度条：承载完成百分比和节点进度。
- 甘特图/时间线：承载计划、依赖和期限。
- 步骤条：承载处理流程和验收状态。
- 预警清单：承载逾期、阻塞、高优先级、无更新任务。
- 效果对比卡：承载执行前后指标或状态验证。

在这个报表类型中，任务、责任、期限、状态和证据比传统分析图表更重要。

## 闭环验证

始终区分“任务完成”和“问题解决”。可验证的结果包括：

- 关联异常已关闭。
- 关联指标改善或达标。
- 项目回到计划。
- 逾期回款追回。
- 成本或费用降低。
- 投诉解决且未复发。
- 巡检问题通过验收。
- 绩效差距缩小。
- 源数据被修正。
- 业务负责人接受结果。

如果任务完成但问题未解决，应复开、创建跟进任务或升级。

## 治理与追溯

- 每个任务必须有来源、目标、负责人、期限、优先级和验收标准。
- 每次状态变化要有时间、操作人和必要评论。
- 延期要有原因、新期限和审批规则。
- 关闭或验收要有证据或验收意见。
- 驳回要说明原因和下一步。
- 批量操作要受权限控制并记录日志。
- 通知和催办要避免噪声，但对逾期和高风险任务及时升级。

## 与其他报表的边界

- 任务来源可能来自 `status-overview-report-design`、`analysis-diagnostic-report-design`、`anomaly-monitoring-report-design`、`performance-evaluation-report-design` 或 `review-recap-report-design`。
- 需要查具体记录时，用 `detail-query-report-design`。
- 需要核对数据正确性时，用 `reconciliation-traceability-report-design`。

## 独立交付与后续接口

- 本 skill 自行输出业务问题、内容块建议、关键字段、筛选器、交互入口和验收口径，保证不依赖其它 skill 也能形成可评审的报表设计。
- 页面结构、组件选择和视觉约束在本 skill 中给出最低可落地建议；需要更细的原型布局时，把这些结果作为后续输入。
- 后续衔接仅作为交付接口：进入原型或实现时，可将本 skill 的输出交给原型、布局或组件流程继续细化，本 skill 不以它们为前置依赖。
## 独立调用最低落地门禁

最终输出至少补齐：

- 数据契约：指标字段、行粒度、基准/阈值、公式、更新时间、数据源或接口依赖。
- 子类型门禁：必须声明唯一 `operationalExecutionSubtype`、执行对象、执行动作和闭环机制。
- 任务模型门禁：必须写清任务来源、记录粒度、责任人、协同人、验收人、优先级、截止时间/SLA、状态流转、上下文信息和处理表单。
- 闭环门禁：必须定义关闭标准、证据要求、复核/验收机制、复开条件和效果验证；不得只统计任务完成率。
- 协同升级门禁：复杂任务必须明确主责、协同、监督、验收和升级路径，避免多人协同无主责。
- 筛选与交互：主筛选、影响组件、下钻/抽屉/跳转参数、返回和重置行为。
- 权限与状态：组织/对象权限、导出权限、空态、加载、错误、无权限、筛选后失效状态。
- 校验方式：任务来源、责任人、期限、进度、证据、验收结果和业务效果是否能形成闭环。
- 后续接口：如进入原型或实现，输出当前 skill 的字段、筛选、交互、组件建议、验收口径和未决问题；后续流程可消费这些信息，但不是本 skill 的前置条件。

## 输出格式

当用户要求设计运营执行类报表时，按这个结构输出：

1. 设计定位：说明执行场景、用户、任务来源和管理目标。
2. 子类型判断：声明唯一 `operationalExecutionSubtype`，并说明为什么不是其他子类型。
3. 思路逻辑：说明问题如何转任务，任务如何推进闭环。
4. 任务模型：说明字段、来源、记录粒度、责任、期限、状态、进度、上下文、处理表单、证据和验收。
5. 优先级与状态流：说明排序规则、SLA、状态转换、关闭标准、复开条件和权限。
6. 布局框架：说明概览、重点任务、分布、进度、风险、结果、闭环层级。
7. 组件映射：说明看板、任务表、甘特图、步骤条、预警清单和效果验证的角色。
8. 交互路径：说明任务抽屉、状态流、更新、附件、转派、催办、延期、提交、验收、驳回和跳转。
9. 派发协同：说明主责、协同、监督、验收、通知和升级机制。
10. 闭环验证：说明如何验证业务问题解决，而不只是任务提交。
11. 权限治理：说明角色、操作权限、日志、通知、升级和审计规则。
12. 执行结论：说明完成、逾期、阻塞、责任和下一步推动动作。

## 质量清单

- 每个任务是否有来源、目标、负责人、期限、优先级、状态和验收标准。
- 是否选择了唯一 `operationalExecutionSubtype`。
- 是否明确一行记录代表什么。
- 执行动作是否明确，用户知道下一步做什么。
- 状态流是否支持分派、执行、提交、验收、驳回、关闭和复开。
- 任务抽屉是否支持进度、备注、附件、转派、催办、延期、提交和验收。
- 逾期、阻塞、高优先级和领导关注任务是否容易找到。
- 是否验证业务问题是否真正解决。
- 是否能承接异常、分析、复盘、流程或人工来源，并回流到绩效、复盘或追溯。
- 日志、证据和状态转换是否可审计。
- 结论是否说明完成情况、卡点责任、业务结果和下一步。

## 避免

- 不要把执行报表做成只看指标的看板。
- 不要把明细表当执行表；没有动作、责任、期限、状态和关闭标准时应归入明细查询。
- 不要只统计任务完成率而不验证业务效果。
- 不要创建没有责任人、期限、来源或验收标准的任务。
- 不要允许重要任务无证据关闭。
- 不要让逾期和阻塞任务隐藏在汇总数字里。
- 不要多人协同时没有主责任人、验收人和升级路径。
